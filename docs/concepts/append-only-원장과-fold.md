# append-only 원장과 fold

**한 줄:** 총점을 변수에 저장하고 덮어쓰는 대신, 사건(기입)을 기록만 계속 쌓고 필요할 때 처음부터
접어서(fold) 현재 값을 계산한다. 은행 통장이 이것 — 잔액이 아니라 입출금 내역을 쌓고 더한다.

## 왜 필요한가 — 문제부터

가장 순진한 채점: 조마다 점수 변수 하나 두고 라운드마다 더한다.

```ts
teamScore[3] += 10;   // 3조 정답, +10
```

동작한다. 그런데 이 프로젝트의 세 요구가 이걸 무너뜨린다.

**1. 되감기 (콘솔 필수기능: "라운드 스킵 / 되감기").** 3번 문제가 오심이라 무효로 돌려야 한다.
`teamScore[3] -= 10`? 그 사이 다른 라운드가 여러 번 덮어써서 **그때 +10이었는지도 모른다** — 배수
토글이 2배였으면 +20이었다. 되돌리려면 라운드마다 얼마였는지를 어딘가 따로 쌓아둬야 하는데, 그럼
변수 하나로는 이미 부족하다.

**2. 크래시 (원칙 4: "프로세스가 죽어 점수가 날아가는 게 실제 시나리오다").** `teamScore`는 메모리
변수다. 노트북이 죽으면 숫자가 사라진다. 살리려면 매 변경마다 저장하고 재시작 때 정확히 복원하는
코드를 따로 짜야 한다.

**3. 연출 (성공 기준: 오디오).** 숫자만 슬그머니 바뀌면 관객은 이유를 못 본다 — 버그처럼 보이고
소리가 안 난다. "3조 무효!"가 화면에서 점수가 내려가는 **사건**이어야 야유가 나온다.

세 문제의 공통 뿌리: **총점을 저장하면 그 총점을 되돌릴 수도, 되살릴 수도, 보여줄 수도 없다.**

## 답: 총점을 저장하지 말고, 사건을 쌓고 접는다

총점을 버린다. 대신 일어난 일(기입)만 순서대로 쌓는다. `ledger.ts`의 기입 4종:

```ts
export const LedgerEntry = z.discriminatedUnion('kind', [
  SeedEntry,   // 낮 PG 점수 — set (마지막 것만 유효)
  RoundEntry,  // 라운드 확정 — add
  AdjustEntry, // 수동 보정 — add
  VoidEntry,   // 되감기 — 이전 기입 하나를 "무효" 표시
]);
```

`UPDATE`도 `DELETE`도 없다. 오직 **append**. 총점은 저장하는 값이 아니라 이 배열을 접어서 그때그때
**파생**하는 값이 된다 ([[파생-상태와-단일-진실-공급원]]). 배열이 유일한 진실이고 총점은 그 그림자다.

(`LedgerEntry`가 `kind`로 갈리는 [[판별-유니온]]인 게 우연이 아니다 — 접을 때 `kind`마다 다르게
처리해야 하고, 그 분기를 컴파일러가 강제한다.)

## fold가 뭔데 — JS의 reduce다

스텁이 남긴 질문이다. **fold**(접기)는 리스트를 훑으며 값 하나로 뭉치는 것이고, JS에선 `reduce`가
바로 이것이다:

```ts
[1, 2, 3].reduce((acc, n) => acc + n, 0);   // 0 → 1 → 3 → 6
```

셋이 필요하다: **누산기**(acc, `0`에서 시작), **합치는 함수**(`acc + n`), **씨앗**(초기값 `0`).
리스트를 왼쪽부터 훑으며 누산기를 갱신하고, 끝의 누산기가 답이다.

원장의 총점도 정확히 이 모양이다. 누산기가 숫자 하나가 아니라 `{조: 점수}` 맵이고, 리스트가 기입
배열일 뿐이다.

## 이 프로젝트에서

`ledger.service.ts:72` `totals()` — "이 함수가 채점 모델 전부다". 접는 규칙 세 줄:

```
1. SEED           : 마지막 것만 유효 (set) — 씨앗이 된다
2. ROUND / ADJUST : 전부 더함 (add)
3. VOID 당한 seq는 건너뜀
```

```ts
totals(teams) {
  const voided = new Set();
  for (const e of this.entries) if (e.kind === 'VOID') voided.add(e.voidsSeq);
  // SEED는 set이라 마지막 것만 — 이게 add의 출발점(씨앗)이 된다
  let seed = {};
  for (const e of this.entries) if (e.kind === 'SEED') seed = e.totals;
  const out = /* 씨앗을 복사해서 시작 */;
  for (const e of this.entries) {
    if (voided.has(e.seq)) continue;                       // 무효는 건너뜀
    const deltas = e.kind === 'ROUND'  ? e.appliedDeltas
                 : e.kind === 'ADJUST' ? e.deltas : null;
    if (deltas) /* out에 더한다 */;
  }
  return out;
}
```

`Date.now()`도 난수도 없다 — 그래서 **같은 원장이면 언제 접어도 같은 총점**이다. 이게 지난 노트가
남긴 "나머지 절반"이다: `totals()`가 인스턴스 필드 `this.entries`를 읽는데도 결정적이어도 되는
이유는, 원장이 append-only라 한 번 쌓인 입력이 두 번 다시 안 변하기 때문이다 ([[순수-함수와-결정성]]).

여러 번 훑으니 코드는 for 루프지만, 하는 일은 기입 배열 → 총점 맵으로 **접는** 것 — fold다.
`ledger.ts` 주석도 그렇게 부른다: *"배열을 처음부터 훑어서 값 하나로 '접는' 걸 fold라고 한다."*

### SEED만 set, 나머지는 add — 비대칭은 왜

두 번째로 남긴 질문. 사회자가 7:58에 낮 PG 점수를 입력하다 오타를 낸다. 만약 SEED도 다른 기입처럼
add였다면, 고치려면 "50 넣었는데 40이었으니 -10 기입"을 **8시 정각에 암산**해야 한다. 그래서 SEED만
set이다 — `totals()`가 `for SEED: seed = e.totals`로 **마지막 것만** 읽으니, 올바른 총점을 그냥 다시
입력하면 최신 SEED가 이전 걸 덮는다. 사람이 시간에 쫓기며 손으로 치는 유일한 기입이라 거기만
편의를 산 것이다. (`ledger.ts` SeedEntry 주석: "마지막 SEED만 유효하다.")

이름의 우연이 아니다: 낮 PG 총점(SEED)이 라운드 델타가 얹히는 **출발점**, 즉 fold의 씨앗 그 자체다.

### 세 문제가 어떻게 풀렸나

- **되감기 = 삭제가 아니라 기입.** "3번 무효"는 `VoidEntry{voidsSeq: 3}` 하나를 **더한다.** 원본
  RoundEntry는 남고, 접을 때 `voided.has(e.seq)`로 건너뛴다. 무효를 무효로 되돌리는 것도 그냥 또
  하나의 기입 — 그래서 `VOIDABLE_KINDS`는 `ROUND | ADJUST`뿐이고 VOID의 VOID는 금지다(다른 길이 있으니).
- **복구 = 다시 접기.** `ledger.service.ts:46` `onModuleInit`이 재시작 때 DB에서 배열만 되살린다.
  복구 코드랄 게 없다 — `totals()`가 아무것도 모른 채 그대로 돈다. 복구되는 건 점수뿐, 진행 중이던
  라운드는 사회자가 20초에 다시 제시한다(값어치 없는 복잡도라 안 되살린다).
- **연출 = 기입이 곧 사건.** VOID가 배열의 한 줄이라, 빔이 "3번 문제 무효!"를 띄우고 점수가 내려가는
  걸 보여줄 수 있다. append-only는 엔지니어링 선택이자 연출 선택이다.

## 언제 쓰고 언제 안 쓰나

이 패턴의 일반 이름이 **이벤트 소싱(event sourcing)** 이다 — 상태를 저장하지 말고 상태를 바꾼
사건들을 저장하라. Redux(액션 로그), git(커밋 로그), 은행 원장이 전부 이것.

**쓴다:** 되돌리기·감사(누가 언제 왜)·재현·복구가 중요한 곳. 이 프로젝트는 넷 다 해당한다.
`AdjustEntry.reason`이 필수인 것도 같은 정신 — 원장이 나중에 "왜 3조가 +10 받았지"에 답하는 이야기가
되게 한다.

**안 쓴다 / 남용:**
- **비싸질 수 있다.** 매번 처음부터 접으면 기입 수에 비례해 느려지고(O(n)), 로그는 무한히 자란다.
  큰 시스템은 스냅샷(중간 총점을 가끔 저장)으로 접는 구간을 줄인다. 이 프로젝트는 하룻밤·조 6개·기입
  수백 건이라 그냥 매번 접는다 — 최적화가 복잡도 값을 못 한다.
- **append는 멱등이 아니다.** 커밋을 두 번 누르면 기입이 둘 쌓여 점수가 두 번 들어간다. 그래서 Live
  커밋은 따로 멱등 장치가 필요하다 ([[멱등성]]) — 원장이 공짜로 주는 성질이 아니다.
- **모든 상태를 원장에 넣지 마라.** 되돌릴 일도 없고 남길 이야기도 없는 순간 상태(입력 중인 폼, 현재
  phase)까지 기입으로 만들면 로그가 소음으로 찬다. 확정됐고·되돌릴 수 있어야 하는 것만 원장에.
