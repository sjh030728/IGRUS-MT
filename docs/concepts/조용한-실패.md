# 조용한 실패 (silent failure)

**한 줄:** 뭔가 틀렸는데 아무것도 안 터지는 것. 에러도 로그도 빨간 줄도 없다.
**살아남는 버그는 거의 전부 이거다.**

## 왜 필요한가 — 이 레포의 버그를 세보면

이 프로젝트에서 지금까지 이름이 붙은 버그들이다. 오른쪽 칸을 보라.

| 버그 | 뭐가 터졌나 |
|---|---|
| `DISPLAY_MODE`가 4개 모드 중 1개만 받았다 (`0005`) | 아무것도 |
| `Game` 타입에 **어떤 게임도 대입할 수 없었다** (`0009`) | 아무것도 — `verify` 23개 전부 통과 |
| `reveal`이 빔에 도달할 경로가 아예 없었다 (`0009`) | 아무것도 |
| README "유일하게 채워진 노트"가 어느새 6개가 됐다 (`0008`) | 아무것도 |
| `grep "#스텁"`이 채워진 노트를 스텁으로 셌다 | 아무것도 — 숫자만 틀림 |
| `awk 'FNR==3'`이 프론트매터가 생기자 0을 반환했다 | 아무것도 — **0이 정답처럼 보인다** |
| `tsx -e`가 개행이 들어가자 아무것도 안 했다 | 아무것도 — 출력 0줄 |

**에러 메시지 총합: 0개.**

### 이건 우연이 아니다

`endsAt + undefined = NaN`은 나자마자 화면이 깨진다. 5초 만에 고친다.
**그런 건 "버그"로 기억되지도 않는다.** ADR이 안 써지고 findings에도 안 올라간다.

**살아남아서 이름이 붙는 버그는 정의상 조용한 것들이다.** 생존 편향이다 —
시끄러운 버그는 표본에 남을 시간이 없다.

그러면 "버그를 줄인다"는 실질적으로 **"조용한 걸 시끄럽게 만든다"**가 된다.
`0003`이 이 프로젝트의 설계 전체를 한 줄로 요약한다:

> 조용히 망하는 것들은 타입으로 막아야 한다. **시끄럽게 망하는 건 어차피 테스트가 잡는다.**

### 왜 이 프로젝트가 유독 이걸 신경 쓰나

보통 웹서비스면 프로덕션 버그가 나도 핫픽스를 배포한다. 여긴 다르다.

- **49분이다.** 8시 35분에 나는 버그는 고칠 시간이 없다. 롤백도 없다
- **리뷰어가 없다.** `0003`: "주석 + 코드 리뷰 — 리뷰어가 없다. 1인 개발이다"

**사람이 잡을 수 있는 층이 없다.** 그래서 기계가 잡아야 하고, 기계는 시끄러운 것만 잡는다.

## 이 프로젝트에서 — 서로 다른 걸 가르치는 셋

### 1. 타입이 통과하는데 뜻이 틀렸다 — `DISPLAY_MODE` (`0005`)

```ts
z.object({ c: z.literal('DISPLAY_MODE'), mode: DisplayState.options[0].shape.mode }),
```

`options[0]`은 "어떤 모드든"처럼 **읽히지만** 실제론 "첫 번째 멤버"다. 그리고 첫 멤버가
`BLACK`이다. 그래서 `{ mode: 'AWARD' }`가 조용히 거절됐다.

타입 검사는 완벽히 통과한다. **`options[0]`이 무슨 뜻인지는 타입이 안 본다** — 자리에
맞는 모양이 왔는지만 본다. `0005`의 마지막 줄:

> 이 버그는 계약이 **자기 자신을 참조**해서 났다. `options[0]`은 "어떤 모드든"처럼
> 읽히지만 실제론 "첫 멤버"다. **유니온 순서를 바꾸면 의미가 조용히 바뀌는 코드였다.**

**교훈: 타입 통과 ≠ 뜻이 맞음.** 그리고 이건 [[타입-vs-런타임-검증]]과 다른 층이다 —
거기선 "런타임에 이상한 값이 온다"가 문제였고, 여기선 **정상값이 조용히 거절**된다.

### 2. 안 쓰이는 코드는 안 틀린다 — 제네릭 `never` (`0009`)

```ts
Game = LockRevealGame<never, never, never>   // 어떤 게임도 대입할 수 없는 타입
```

컴파일된다. `npm run verify` 23개도 전부 통과했다. **쓰는 사람이 없어서.**

> 셋 다 **첫 소비자가 도착한 순간에만** 드러나는 종류다. 타입을 아무리 노려봐도 안 보이고,
> `npm run verify` 23개도 전부 통과했다. (`0009`)

그리고 `0009`의 진짜 결론이 여기 있다:

> `0001`이 산 것은 "계약이 완벽하다"가 아니라 **버그를 만나는 자리**다.

계약을 먼저 정의한 게 버그를 **없앤** 게 아니다. 30줄짜리 더미를 쓰다 만나게 했고,
고치는 데 20분 걸렸다. 나중에 추출했으면 단계 2에서 진짜 퀴즈를 짜다 만났을 거고,
**그땐 게임 코드를 의심했을 것이다** — 계약이 원인인 줄 모른 채.

**같은 버그인데 만나는 자리에 따라 20분이 되기도 하고 하루가 되기도 한다.**

### 3. "없다"와 "안 돌았다"는 같은 화면이다 — 내 재현 명령

`findings.md`에 재현 명령을 적었는데 안 돌았다. 둘이 겹쳤다:

- `tsx -e`에 개행이 들어가면 **에러 없이 아무것도 안 한다.** 출력 0줄
- 상대경로 `./src/index.js`가 `-e` 컨텍스트에선 안 잡힌다

돌렸더니 출력이 0줄이었다. **그걸 "버그가 없네"로 읽을 뻔했다.**

**조용한 실패의 최악은 실패처럼 안 보이는 게 아니라 성공처럼 보이는 것이다.**
`awk 'FNR==3'`도 같았다 — 프론트매터가 생기면서 죽었는데, 반환값이 `0`이라
"스텁이 0개"인지 "명령이 죽었는지" 화면이 똑같다.

**교훈: 0을 봤으면 0이 나오는 걸 확인해봐야 한다.** 안 되는 걸 넣어서 안 되는 게
나오는지 보는 것 — 그게 이 두 개를 잡은 방법이다.

## 어디서 시끄러워지나 — 층

| 층 | 언제 | 이 프로젝트 |
|---|---|---|
| **컴파일 타임** | 저장하는 순간 | 타입, `never` [[빠짐없음-검사와-never]] |
| **계약 검증** | `npm run verify` | `verify.ts` |
| **첫 소비자** | 실제로 써볼 때 | `dummy.game.ts`가 `0009`의 셋을 다 찾아냈다 |
| **리허설** | 단계 5 | 뒷줄 가독성, 타임라인 실측 |
| **당일 8시 35분** | 40명 앞에서 | **여기서 나면 진 것** |

**위로 갈수록 싸다.** 그리고 이 프로젝트의 설계 결정 대부분이 버그를 위로 밀어올리는
장치다 — `0001`(계약 먼저), `0003`(원칙을 타입으로), `verify.ts`, 단계 1의 더미.

CLAUDE.md가 "단계 5에 도달하는 게 목표"라고 한 것도 같은 얘기다. 리허설은 마지막 기회고,
거기서도 못 잡으면 그 밤에 40명 앞에서 잡힌다.

## 언제 쓰고 언제 안 쓰나

### ★"전부 시끄럽게"는 틀렸다★

시끄러움은 그 자체로 좋은 게 아니다. **언제, 누구한테 시끄러운지가 전부다.**

**시점을 틀리면 방어가 사고가 된다.** `0003`이 런타임 assert를 반려한 이유:

> 8시 35분에 assert가 터지면 그건 방어가 아니라 **사고**다.
> 컴파일 타임에 막아야 무대에 안 오른다.

버그를 시끄럽게 만들었는데 그 시끄러움이 무대 위에서 나면, 조용한 것보다 나쁠 수 있다.

### 누구한테 시끄러울지 — `anticheat.ts`가 셋 다 다르게 판단한다

**(a) 참가자한테는 일부러 조용하다.** 탭 레이트 상한을 넘은 탭은 거절이 아니라 그냥 버린다:

> 폰에 에러를 띄우면 정상 참가자가 자기가 잘못한 줄 알고 **손을 멈춘다.**
> 조용히 상한선까지만 인정한다.

시끄럽게 하면 오디오가 죽는다. 성공 기준 위반이다.

**(b) 그런데 점수를 조용히 깎는 건 금지다.**

> 앱이 조용히 점수를 깎으면 아무도 이유를 모른 채 **결과만 이상해진다.**

같은 파일에서 하나는 조용해야 하고 하나는 안 된다. **차이는 "결과에 남느냐"다.**
버려진 탭은 흔적이 없지만 깎인 점수는 순위를 바꾼다.

**(c) 그리고 시끄러움을 사람한테 넘긴다.**

> 앱이 하는 일: 사회자한테 "3조 김철수, 15.0tap/s 25초 유지, 간격 편차 2ms"를 보여준다.
> 사회자가 하는 일: 마이크 잡고 "야 김철수 손 들어봐."
> **★후자가 압도적으로 재미있고, 그게 성공 기준이다.★**
> 치팅 적발이 콘텐츠가 된다. 앱이 처리하면 **그냥 조용한 에러 메시지다.**

앱이 시끄럽게 하면 에러고, 사회자가 시끄럽게 하면 콘텐츠다.

**정리: 조용함을 고칠 때 고르는 건 "시끄럽게 할까"가 아니라 "누구한테, 언제"다.**
개발자한테 컴파일 타임에 → 최고. 사회자한테 콘솔에서 → 좋다.
참가자한테 폰에서 → 손이 멈춘다. 관객한테 8시 35분 빔에서 → 진 것이다.

### 일반적인 남용 지점

- **에러를 삼키는 `try/catch`** — 조용한 실패를 **직접 만드는** 코드다. 잡았으면 뭘 하든가
  다시 던지든가
- **`catch {}` 안의 기본값** — `?? []`가 "없음"인지 "실패"인지 구별을 없앤다.
  위 3번이 정확히 그 모양이다
- **경고를 로그로만** — 아무도 로그를 안 본다. `0003`: "주석은 안 읽힌다. 컴파일 에러는
  안 읽을 수가 없다"
