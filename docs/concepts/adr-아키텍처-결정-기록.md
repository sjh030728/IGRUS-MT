# ADR (아키텍처 결정 기록)

**한 줄:** "왜 이렇게 짰나"를 결정 하나당 파일 하나로 남기는 것. 번호가 붙고, 한 번 쓰면
고치지 않는다. Architecture Decision Record의 약자.

## 왜 필요한가

6주 뒤. 단계 4쯤 와서 콘솔을 짜다가 Zod가 거슬린다. 스키마 쓰는 게 귀찮고, Nest 튜토리얼은
전부 `class-validator`를 쓴다. "그냥 Nest 방식으로 바꿀까?"

그 순간 필요한 정보는 이거다: **누가 이미 이걸 검토했나? 왜 안 골랐나?**

코드를 열면 `z.object(...)`가 있다. **"무엇"은 있는데 "왜"가 없다.** 그래서 다시 검토한다.
반나절 태우고 같은 결론에 도달한다. 아니면 **다른 결론에 도달해서 갈아엎고, 3주 뒤에
원래 이유에 부딪힌다.**

여기서 "누가"는 남이 아니다. **6주 뒤의 본인이다.** 1인 개발이면 특히 그렇다 —
물어볼 사람이 없다.

### git log가 있잖아

```
8f20148 add project context
```

이게 이 레포의 실제 커밋 메시지다. 커밋은 **무엇을 바꿨는지**를 적고, 왜 그 대안을 버렸는지는
안 적는다. 적는다 해도 `git log`를 뒤져서 6주 전 결정을 찾아내는 사람은 없다.

### 코드 주석에 쓰면 되잖아

`docs/README.md`가 이 질문에 답한다:

> **기준: 답이 파일 2개 이상에 걸치는가.**
> - "왜 `phaseEndsAt`이 nullable이냐" → 한 곳. **코드 주석.**
> - "왜 Zod냐" → 계약 전체. **ADR.**

"왜 Zod냐"를 주석으로 쓰려면 **어느 파일에 쓰나?** `events.ts`? 그럼 `display.ts` 보는
사람은 못 읽는다. 전부에 쓰면 6벌이 되고 5벌이 뒤처진다.

**갈 곳이 없는 이유들이 있다. 그게 ADR이 사는 곳이다.**

## ADR이 어떻게 생겼나

이 레포의 `docs/decisions/0002-와이어-검증은-zod-단일정의로-한다.md`가 실물이다:

```md
# 0002. 와이어 검증은 Zod 단일 정의로 한다

- 날짜: 2026-07-15
- 상태: 확정
- 개념: [[타입-vs-런타임-검증]], [[스키마-기반-타입-추론]], [[판별-유니온]]

## 무슨 결정인가
## 왜
## 안 고른 것
## 이 결정이 사는 곳
```

**형식은 표준이 없다.** 원래 제안(Michael Nygard, 2011)은 맥락/결정/결과 3단이었고,
팀마다 고쳐 쓴다. 위는 이 프로젝트가 정한 것이다. 중요한 건 칸 이름이 아니라 아래 셋이다.

### 1. 번호가 붙는다

`0001`, `0002`, `0003`, `0004`. **순서가 정보다.** 0004는 0001을 읽고 내린 결정이다.
나중에 0007이 0002를 뒤집을 수도 있는데, 그때 번호가 그 관계를 표현한다.

### 2. ★한 번 쓰면 고치지 않는다★

여기가 제일 자주 오해받는 부분이다. 결정이 바뀌면 **기존 ADR을 수정하는 게 아니라**,
새 ADR을 쓰고 옛날 것의 상태를 바꾼다:

```
- 상태: 폐기됨 (0007로 대체)
```

옛날 ADR의 본문은 **그대로 둔다.** 틀린 판단이었어도 지우지 않는다. 그때 그렇게 생각한
이유가 다음 판단의 재료이기 때문이다.

**이거 어디서 본 구조 아닌가?**

> 되감기가 공짜다. "3번 문제 무효" = 삭제가 아니라 VOID 기입을 하나 더 추가.
> 원본은 남는다. (`packages/protocol/src/ledger.ts`)

**`LedgerEntry`의 `VOID`와 정확히 같은 패턴이다.** 지우지 않고 무효 기입을 추가한다.
ADR 폴더는 append-only 원장이고, `상태` 필드가 `VOID`에 해당한다.
같은 아이디어가 점수와 문서 양쪽에 나온 것이다 → [[append-only-원장과-fold]]

(그래서 `- 상태: 확정` 같은 필드가 있는 것이다. 파일을 지울 거면 상태 필드가 필요 없다.)

### 3. "안 고른 것"이 본체다

이게 없으면 ADR이 아니라 그냥 설명서다. 0002를 보면:

> **Nest class-validator DTO** — Nest 관용이라 서버만 보면 자연스럽다. 하지만 데코레이터
> 기반이라 브라우저(빔/폰)에서 재사용이 어렵고, 그러면 빔·폰이 타입을 따로 들게 된다.
> **계약이 두 벌 나는 것** — 0001을 무효화한다.

6주 뒤에 "Nest 방식으로 바꿀까?" 하는 순간 **이 문단이 답이다.** 재검토가 반나절에서
30초가 된다. 그리고 "0001을 무효화한다"에서 번호가 일을 하는 게 보인다.

`docs/README.md`의 나머지 기준:

> 그리고 **누가 다르게 했을 법한 것**만 쓴다. "for문을 썼다"는 결정이 아니다.

**"안 고른 것"을 못 쓰겠으면 그건 결정이 아니었다는 뜻이다.** 대안이 없었으면 판단한 게 아니다.

## 이 프로젝트에서

`docs/decisions/`에 4개 있다. `docs/README.md`가 폴더 두 개를 나눈 이유를 설명한다:

| | `decisions/` (ADR) | `concepts/` |
|---|---|---|
| 뭐가 | **왜 이렇게 짰나** | **여기 쓰인 개발 지식** |
| 범위 | 이 프로젝트 한정 | 어디서든 |
| 날짜 | 있음 | 없음 |
| 유통기한 | MT 끝나면 죽음 | 5년 뒤에도 씀 |
| 누가 | 개발 세션이 자동으로 | 학습 세션이 질문 받고 |

**지금 읽고 있는 이 파일은 `concepts/`에 있다.** "ADR이 뭐냐"는 다음 프로젝트에서도 쓰는
지식이니까. 반면 "이 프로젝트가 Zod를 골랐다"는 `decisions/0002`다.
[[판별-유니온]]과 `0002`가 짝인 것과 같은 구조다 — 개념과 결정이 서로를 링크한다.

`0004-1인-개발로-게임을-4개로-줄인다.md`의 헤더가 이 분리를 잘 보여준다:

```
- 개념: (없음 — 순수 범위 판단이라 개발 지식이 안 들어간다)
```

**ADR인데 딸린 개념이 0개일 수 있다.** 게임을 4개로 줄인 건 개발 지식이 아니라 범위 판단이다.
둘이 직교(orthogonal)라는 증거다.

### 자동으로 쌓이게 해놨다

CLAUDE.md:

> **사용자가 기억해서 명령을 쳐야 하면 그 트랙은 죽는다.** 설문이 죽는 것과 같은 이유다.
> 세션은 깔끔하게 안 끝난다 — 피곤해서 노트북을 덮는다. 그때 재료가 이미 남아 있어야 한다.

ADR의 실패 원인 1위는 형식이 아니라 **아무도 안 쓰는 것**이다. 쓰기로 해놓고 3주 뒤부터
빈다. 그래서 개발 세션이 시키지 않아도 남기게 해뒀다.

## 언제 쓰고 언제 안 쓰나

### 쓴다 — 두 조건을 **둘 다** 통과할 때

1. **답이 파일 2개 이상에 걸치는가** (한 곳이면 코드 주석)
2. **누가 다르게 했을 법한가** (대안이 없었으면 결정이 아니다)

거기에 하나 더 얹으면 유용하다: **되돌리기 비싼가.**
DB를 고르는 건 3주 뒤에 못 바꾼다. 함수 이름은 5분이면 바꾼다. 되돌리기 싼 결정은
ADR을 쓰는 시간이 그냥 바꾸는 시간보다 길다.

### 안 쓴다 — 남용 지점

CLAUDE.md가 직접 경고한다:

> 해당 없으면 **안 쓴다.** 세션마다 하나씩 짜내면 아무도 안 읽는 폴더가 된다.

- **코딩 컨벤션** (들여쓰기 2칸, 세미콜론) → linter가 강제한다. 문서는 안 읽힌다
- **되돌리기 싼 것** → 그냥 바꿔라
- **대안이 없었던 것** → "TypeScript를 썼다"는 결정이 아니다
- **개인 취향** → 안 고른 이유가 "안 좋아해서"면 쓸 게 없다
- **채우기 위해 쓰는 것** ← 이게 제일 흔한 실패다. 20개 중 15개가 시시하면 나머지 5개도 안 읽힌다

**ADR 폴더의 가치는 개수에 반비례한다.** 4개짜리 폴더는 30분이면 다 읽고 프로젝트 전체의
판단 구조가 머리에 들어온다. 40개면 아무도 안 연다.
