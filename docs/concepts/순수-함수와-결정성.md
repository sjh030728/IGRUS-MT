# 순수 함수와 결정성

**한 줄:** 같은 입력이면 항상 같은 출력을 내고(결정적), 그 출력 말고는 바깥 세상에 아무
흔적도 안 남기고 바깥의 변하는 값도 안 읽는(순수) 함수. 재현·복구·되감기가 전부 이 위에 선다.

## 왜 필요한가 — 문제부터

CLAUDE.md 원칙 4: **"프로세스가 죽어 점수가 날아가는 게 실제 시나리오다."** 그래서 확정 점수는
원장(DB)에 쌓고, 재시작하면 원장을 처음부터 다시 훑어 점수를 되살린다(fold — [[append-only-원장과-fold]]).
복구 코드랄 게 따로 없다. `ledger.service.ts:46`:

```ts
async onModuleInit() {
  this.entries = await this.db.loadLedger(this.sessionId);   // 배열만 되살린다
  // → totals()는 아무것도 모른 채 그대로 돈다
}
```

여기 조용한 함정이 있다. 점수를 되살리는 그 함수가 만약 **지금 시각**을 보거나(`Date.now()`),
**난수**를 쓰거나(`Math.random()`), **바깥의 변하는 값**을 읽는다면 — 죽기 전과 재시작 후의 점수가
달라진다. 8시 35분에 3조 점수가 이유 없이 바뀐다. 되감기("3번 문제 무효")도 같은 함수를 다시 돌리는
거라, 결과가 그때그때 다르면 재현이 안 된다.

그래서 필요한 보장은 하나다: **같은 원장 → 반드시 같은 점수.** 이 보장의 이름이 **결정성**이고,
그걸 확실히 얻는 방법이 **순수 함수**다. 이 프로젝트에서 이건 좋은 습관이 아니라 원칙 4의 전제다.

## 결정적 ≠ 순수 — 다른 두 성질이다

자주 같은 말처럼 쓰지만 가리키는 곳이 다르다.

- **결정성(deterministic)** 은 *출력*에 대한 성질이다. 같은 입력 → 항상 같은 출력.
- **순수성(pure)** 은 *부작용*에 대한 성질이다. ① 바깥을 안 바꾼다(DB 쓰기·전역 변경·소켓
  emit·로그) ② 바깥의 변하는 값을 안 읽는다(시각·난수·전역·DB). 오직 인자만 보고 반환값만 낸다.

**부작용(side effect)** = 함수가 반환값 말고 남기거나 끌어오는 모든 것. 순수 함수는 부작용이 0이다.

둘의 관계:

- 순수하면 인자만 보므로 → **자동으로 결정적이다.** (순수 ⟹ 결정적)
- 결정적인데 불순할 수 있다: `function double(a){ console.log(a); return a*2 }` — 반환은 늘 `a*2`(결정적)
  지만 로그를 남긴다(불순).
- 순수한데 비결정적은 사실상 불가능하다. 비결정성의 출처(시각·난수·전역)는 전부 "바깥 읽기"인데
  그게 곧 불순이다.

그래서 `game.ts:218`의 `score()` 주석이 **"순수 + 결정적"** 둘 다 적은 건 중복이 아니라 두 가지를
요구하는 것이다:

```ts
/** LOCKED 진입 시 1회. 순수 + 결정적 (되감기/재시작 때 다시 돌아도 같은 답이 나와야 한다).
 *  ★배수를 보지 못한다★ — 코어가 곱한다. 게임이 배수를 알면 두 곳에서 곱해진다. */
score(round: RoundSpec, bag: SubmissionBag<TAnswer>): ScoreResult;
```

- **결정적**이라서: 되감기·재시작 때 다시 돌려도 같은 델타가 나온다.
- **순수**라서: `score`가 점수를 직접 저장하거나 소켓에 emit하면 안 된다 — 그건 코어(원장 append +
  gateway emit)의 몫이다. 게임은 *계산만*, 적용(부작용)은 코어. "배수를 게임이 곱하면 두 곳에서
  곱해진다"는 경고가 바로 이 정신이다: 부작용은 한곳에서만.

## 이 프로젝트에서

**채점의 심장은 순수하다.** `ledger.service.ts:72` `totals()` — "이 함수가 채점 모델 전부다":

```ts
totals(teams: readonly TeamInfo[]): Record<TeamId, number> {
  const voided = new Set<number>();
  for (const e of this.entries) if (e.kind === 'VOID') voided.add(e.voidsSeq);
  // ... SEED는 마지막 것만, ROUND/ADJUST는 전부 더함, VOID 당한 seq는 건너뜀
}
```

`Date.now()`도 `Math.random()`도 `await`도 없다. 오직 기입 배열을 순서대로 접을 뿐이다. 그래서 위의
`onModuleInit`이 배열만 되살리면 점수판이 죽기 전 그 값 그대로 이어진다.

**스냅샷 투영기도 순수하다.** `projector.ts:20` — 화면을 그리는 계산인데 상태를 안 건드린다:

```ts
// ★순수 함수다. 여기서 상태를 바꾸지 마라★
export function projectDisplay(s: SessionState, board: Scoreboard): DisplayState { ... }
```

상태 하나를 받아 "지금 니 화면 전부"를 돌려준다. 그래서 재접속이 스냅샷 1발로 끝나고, 같은 상태면
빔·폰·콘솔이 늘 같은 그림을 본다 — 이게 [[파생-상태와-단일-진실-공급원]]이 서는 토대다.

### loadRounds는 async인데 순수할 수 있나 — 아니, 그래서 순수를 안 요구한다

스텁이 남긴 질문이다. `game.ts:208`:

```ts
/** ★반드시 결정적일 것. ORDER BY random() 금지★ */
loadRounds(ctx: LoadCtx): Promise<readonly RoundSpec[]>;
```

`Promise`를 반환하고 Postgres(문제은행)를 읽는다. DB 읽기는 **부작용**이다 — 바깥 세상이고, 같은
쿼리가 어제와 오늘 다른 걸 줄 수 있다. 그러니 `loadRounds`는 **순수하지 않다.**

> `async` / `Promise`는 "이 함수가 시간 걸리는 바깥일(DB·네트워크)을 하고 결과가 나중에 온다"는
> 표식이다. 순수 함수는 바깥일을 안 하니 동기(sync)다. 그래서 `score`는 `Promise`가 아니고
> `loadRounds`는 `Promise`다 — **계약이 타입만 보고도 누가 순수 구역이고 누가 부작용 구역인지
> 알려준다.**

핵심은 계약이 `loadRounds`엔 순수를 요구하지 *않고* **결정성만** 요구한다는 것(그래서 콕 집어
`ORDER BY random()` 금지). 왜 결정성만으로 충분한가: 재시작 복구는 `score`를 다시 돌리지 않는다 —
확정 점수는 이미 원장에 있다. 복구가 `loadRounds`에 기대는 유일한 지점은 "진행 중이던 라운드를
사회자가 다시 제시할 때 같은 `roundId`가 나오는 것"뿐이다. DB를 읽는 것 자체(세그먼트 투입 시 1회,
뜨거운 경로 밖)는 문제가 아니고, *난수 정렬*이 문제다.

### totals()는 this.entries를 읽는데 순수한가

엄밀히는 아니다. `this.entries`는 인자가 아니라 인스턴스 필드라, "인자만 읽는다"는 교과서 정의에서
살짝 벗어난다. 그런데 원장은 **append-only**다 — 덮어쓰기도 삭제도 없이 뒤에 쌓기만 한다. 그래서 한
번 자리 잡은 기입은 두 번 다시 안 변한다. 실제로 필요한 성질은 "인자만 읽음"이 아니라 **"원장에 대해
결정적"** 이고, append-only가 그 입력의 안정성을 보장한다. 순수성은 목표가 아니라 결정성을 얻는
수단이었음을 보여주는 자리다 — 자세한 건 [[append-only-원장과-fold]].

## 언제 쓰고 언제 안 쓰나

프로그램은 결국 부작용을 내야 쓸모가 있다. 점수를 화면에 그려야 하고, DB에 저장해야 하고, 소켓으로
쏴야 한다. **전부 순수하게 만들 수는 없다** — 그러면 아무 일도 안 하는 프로그램이다.

요령은 부작용을 *없애는* 게 아니라 **한곳에 모으는** 것이다. 순수한 계산을 가운데 두고(코어), 부작용을
가장자리로 민다(껍질). 흔히 **"함수형 코어, 명령형 껍질"** 이라 부른다. 이 프로젝트의 분할이 그대로 이것:

| 순수 구역 (코어) | 부작용 구역 (껍질) |
|---|---|
| `score` / `parseAnswer` | `LedgerService.append` (원장 쓰기 + DB 미러) |
| `totals` / `scoreboard` | `gateway`의 소켓 emit |
| `projectDisplay` / `projectPlay` / `projectHost` | `loadRounds` (DB 읽기) |

**남용/함정은 부작용을 *숨기는* 것이다.** 순수인 척하면서 몰래 전역을 건드리거나, 캐시를 심거나,
시각을 읽으면 — 결정성이 조용히 깨지고, 그게 드러나는 시점은 재시작한 8시 35분이다([[조용한-실패]]).
차라리 `async`/`Promise`로 "나 부작용 있음"을 타입에 드러내라. `loadRounds`가 정확히 그렇게 한다.

판단 기준 한 줄: **재현·복구·되감기의 대상이 되는 계산은 순수하게, 바깥과 얘기하는 일은 가장자리로.**
