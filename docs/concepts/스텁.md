# 스텁 (stub)

**한 줄:** 껍데기만 만들어두고 알맹이는 나중에 채우는 것. **이름과 자리는 지금 존재하고,
내용은 없다.**

## 왜 필요한가

개발 세션이 `events.ts`를 짜다가 `z.discriminatedUnion`을 처음 쓴다. 사용자는 그게 뭔지
모른다. 두 가지 길이 있는데 **둘 다 나쁘다.**

**(a) 아무것도 안 한다.** 6주 뒤에 사용자는 판별 유니온을 모른다. 더 나쁜 건,
**자기가 그걸 모른다는 것도 모른다.** 모르는 걸 검색할 수는 없다. 이름을 알아야 검색한다.

**(b) 지금 설명을 다 쓴다.** 개발 시간이 문서 쓰는 데 태워진다. 그리고 사용자가 물어보지도
않은 걸 쓰는 거라 겨냥이 안 된다 — 뭘 모르는지 모르는 상태에서 쓴 설명은 대개 빗나간다.
그렇게 나온 문서는 안 읽힌다.

**스텁이 제3의 길이다.** 이름 + 한 줄 + 태그. 30초면 끝난다.

```md
---
tags:
  - 영역/계약-타입
---

# 판별 유니온 (discriminated union)

> **스텁.** `/learn 판별 유니온` 으로 물어보면 채워진다. #스텁

**한 줄:** 공통 필드 하나(판별자)의 값에 따라 나머지 필드가 뭔지 확정되는 타입.
```

30초를 써서 산 것:

- **존재한다** → `[[판별-유니온]]` 링크가 안 깨진다. 다른 노트가 지금 당장 가리킬 수 있다
- **이름이 있다** → 이제 검색할 수 있다. "뭘 모르는지"가 목록이 됐다
- **비어 있고, 그게 보인다** → 태그로 잡힌다. 큐가 된다
- **겨냥된다** → 사용자가 궁금할 때 궁금한 것부터 채워진다

`docs/README.md`가 이 성질을 정확히 짚는다:

> 태그를 쓰는 이유: 스텁도 실제 파일이라 위키링크가 정상 해결된다. 그래서 그래프에선
> 채워진 노트와 구분이 안 된다. 발견은 태그가 맡는다.

### 태그가 두 종류인 게 보일 텐데 — 다른 축이다

- `#스텁` — **채워졌나.** 채우면 뗀다. **큐다** → [[큐]]
- `영역/*` — **뭐에 관한 노트인가.** 채워도 남는다. **지도다**

그래서 채워진 노트엔 `영역/`만 남는다. 지금 읽고 있는 이 노트가 그렇다(맨 위를 보라).

둘이 사는 자리가 다른 것도 같은 이유다. `docs/README.md`:

> **본문 = 사라질 것, frontmatter = 남을 것.** 스텁 태그는 "아직 안 채워졌다"는 안내문에
> 붙어 있고 `/learn`이 채우면서 그 줄을 통째로 지운다. 영역은 채워져도 참이다.

**태그를 떼는 게 아니라 태그가 타고 있던 줄이 통째로 지워진다.** "스텁 해제"라는 별도
동작이 없다 — 채우는 행위 자체가 큐에서 빼는 행위다. 까먹을 수가 없다.

(`영역/`에 뭐가 있는지는 README가 관리한다. 여기 옮겨 적으면 하나 느는 순간 이 노트가
틀린다. 열거는 썩는다.)

**실재하되 비어 있다.** 이 두 개가 동시에 참인 게 스텁의 전부다. 없으면 참조가 깨지고,
채워져 있으면 비용을 이미 낸 것이다. 스텁은 그 사이에 있다.

## 코드에서의 스텁 — 원래 이 단어가 살던 곳

이 프로젝트가 문서에 쓴 건 **빌려온 비유**다. 원래는 코드 용어이고, 다음 프로젝트에선
이쪽으로 만난다.

### 1. 함수 스텁

```ts
function score(round, submissions) {
  throw new Error('not implemented');
}
```

시그니처만 있고 본문이 없다. **컴파일이 된다는 게 핵심이다.** 호출부를 먼저 짤 수 있고,
타입도 맞고, 다만 부르면 터진다.

### 2. 테스트 더블 — 제일 많이 만날 것

테스트에서 진짜 대신 끼우는 가짜를 통틀어 **테스트 더블(test double)**이라고 한다.
스턴트 더블에서 온 말이다. 종류가 넷인데 이름이 자주 섞여 쓰인다:

| 이름 | 뭘 하나 |
|---|---|
| **스텁(stub)** | **미리 정해진 답만 돌려준다.** "DB가 물어보면 이 8문제를 준다" |
| **목(mock)** | 호출됐는지를 **검증한다.** "저장 함수가 정확히 1번 불렸나" |
| **페이크(fake)** | 진짜로 동작하는 간이 구현. 인메모리 DB 같은 것 |
| **스파이(spy)** | 진짜를 부르되 기록도 남긴다 |

**스텁과 목의 차이가 제일 자주 헷갈린다.** 스텁은 **답을 주고**(결과를 검사한다),
목은 **호출을 감시한다**(과정을 검사한다). 스텁을 쓰는 테스트는 "결과가 맞나"를 묻고,
목을 쓰는 테스트는 "이 함수를 불렀나"를 묻는다.

### 3. 타입 스텁

`.d.ts`(TypeScript), `.pyi`(Python). **구현 없이 타입만** 있는 파일. JS 라이브러리에
타입만 얹을 때 쓴다. `@types/node`가 이거다 — 이 레포 `package.json`에 있다.

### 4. RPC 스텁

원격 호출을 로컬 함수처럼 보이게 하는 껍데기. gRPC 클라이언트가 대표적이다.
`client.getUser(1)`처럼 생겼지만 안에선 네트워크로 나간다.

### 넷의 공통점

**껍데기가 진짜 자리에 앉아 있어서, 나머지를 먼저 짤 수 있다.**

문서 스텁도 똑같다. `[[판별-유니온]]`이 그 자리에 앉아 있으니 다른 노트를 먼저 쓸 수 있다.

## 이 프로젝트에서

**단계 1이 코드 스텁으로 굴러간다.** CLAUDE.md:

> **단계 1에 게임이 0개인 건 의도된 것이다.** 더미 문제로 리빌 연출부터 완성한다.

**그 "더미 문제"가 스텁이다.** `apps/server/src/games/dummy.game.ts` — `LockRevealGame`
인터페이스를 구현하되 알맹이는 가짜다. 문제 은행도, 설문 데이터도, 채점 로직도 없이
리빌 루프(잠금 → 카운트다운 → 동시 공개 → 점수 애니메이션)를 **완성**한다.

이게 없으면 단계 1을 할 수가 없다. 게임이 진짜여야 리빌을 만들 수 있다면, 게임을 먼저
만들어야 하고, 그러면 CLAUDE.md가 금지한 순서가 된다:

> 게임 하나를 골라서 서버·빔·폰을 세로로 관통하면, 다음 게임에서 같은 걸 또 짜게 되고
> 코드가 두 벌 나온다.

**스텁이 "레이어로 짠다"를 물리적으로 가능하게 하는 장치다.** 아래층을 가짜로 막아두면
위층을 진짜로 만들 수 있다.

### 스텁을 끼우려면 주입받아야 한다 — 아직 안 정해진 것

스텁은 아무 데나 못 끼운다. **바깥에서 넣어줄 구멍이 있어야 한다.**

`game.ts`를 보면 구멍이 하나 있다:

```ts
export interface LoadCtx {
  segmentId: SegmentId;
  teams: readonly TeamInfo[];
  roster: readonly RosterEntry[];
}

loadRounds(ctx: LoadCtx): Promise<readonly RoundSpec<TPlayPrompt>[]>;
```

`ctx`를 **주입받는다**(바깥에서 넣어준다). 가짜 `ctx`를 넣으면 가짜 조/가짜 명단으로
돌려볼 수 있다.

**그런데 `LoadCtx`에 DB 핸들이 없다.** 주석은 "문제은행(Postgres) 조회는 여기서 끝낸다"고
하는데, 조회할 수단이 `ctx`로 안 들어온다. 그럼 게임 모듈이 DB 클라이언트를 직접 import해야
하고, 그러면 **DB 없이는 게임 모듈을 못 돌린다** — 스텁을 끼울 구멍이 없다.

단계 2에서 정해질 문제다. 버그가 아니라 아직 안 온 결정이다. 다만 여기가
"주입 안 하면 스텁 못 끼운다"의 실물 예시다.

## 언제 쓰고 언제 안 쓰나

### 안 쓴다 — 순수 함수엔 스텁이 필요 없다

`game.ts`의 메서드 3개 중 2개가 순수 함수라고 주석에 못 박혀 있다:

> `parseAnswer` — COLLECT 중 제출마다. **순수 함수.**
> `score` — LOCKED 진입 시 1회. **순수 + 결정적**

순수 함수는 입력만 주면 답이 나온다. 가짜로 대체할 의존성이 애초에 없다. **그냥 부르면 된다.**

스텁이 필요해지는 건 의존성이 있을 때다 — DB, 시계, 네트워크, 랜덤. 그래서 이 계약이
`score()`를 순수로 못 박은 게 테스트 편의까지 같이 산 것이다 → [[순수-함수와-결정성]]

**"스텁이 많이 필요하다"는 설계가 얽혀 있다는 신호인 경우가 많다.**

### 남용 지점 — 스텁이 눌러앉는 것

`throw new Error('not implemented')`가 프로덕션에 나가는 건 실제로 흔하다.
**스텁은 큐지 종착지가 아니다.**

여기서 코드 스텁과 문서 스텁이 갈린다:

| | 안 채우면 |
|---|---|
| **코드 스텁** | 런타임에 터진다. **컴파일은 통과한다** |
| **문서 스텁** | 아무 일도 안 일어난다. 링크는 살아 있다 |

**코드 스텁은 컴파일러를 속인다. 문서 스텁은 아무도 안 속인다.**

그래서 README가 문서 스텁에 대해선 이렇게 말할 수 있는 것이다:

> 밀려 있어도 문제가 아니다. 개발은 안 막힌다.

**코드 스텁엔 절대 저렇게 못 쓴다.** 같은 단어인데 위험도가 다르다.

그리고 이게 이 프로젝트가 스텁에 **태그**를 단 이유다. 문서 스텁은 안 터지니까,
터지지 않는 대신 **보이기라도 해야 한다.** 안 보이는 큐는 큐가 아니다.

### 테스트를 스텁으로 도배하는 것

전부 가짜로 감싸면 테스트는 통과하는데 진짜 코드는 아무도 안 돌린 게 된다.
`verify.ts`가 반대로 간다 — 진짜 Zod 스키마에 진짜 조작 페이로드를 먹인다:

```
OK   n=999999 거절 (배치 상한)
```

가짜가 하나도 없다. 그래서 저 OK가 실제로 뭔가를 보증한다.
