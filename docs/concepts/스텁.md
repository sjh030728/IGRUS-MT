# 스텁 (stub)

**한 줄:** 껍데기만 만들어두고 알맹이는 나중에 채우는 것. **이름과 자리는 지금 존재하고,
내용은 없다.**

## 왜 필요한가

개발 중에 새 개념(예: 판별 유니온)이 등장했다. (a) 아무것도 안 하면 — 사용자는 6주 뒤에도
그걸 모르고, 더 나쁘게는 **모른다는 것도 모른다**(이름을 알아야 검색이라도 한다).
(b) 지금 설명을 다 쓰면 — 개발 시간이 문서에 타고, 묻지도 않은 설명은 겨냥이 빗나가 안 읽힌다.

스텁이 제3의 길이다: 이름 + 한 줄 + 태그, 30초. 그걸로 사는 것 —
**존재한다**(`[[링크]]`가 안 깨진다) / **이름이 생긴다**(검색 가능) / **비어 있는 게
보인다**(태그로 잡혀 큐가 된다) / **겨냥된다**(궁금할 때 궁금한 것부터 채워진다).

## 코드에서의 스텁 — 원래 이 단어가 살던 곳

**1. 함수 스텁.** `function score() { throw new Error('not implemented') }` —
시그니처만 있고 본문이 없다. **컴파일이 된다는 게 핵심.** 호출부를 먼저 짤 수 있고,
다만 부르면 터진다.

**2. 테스트 더블 — 제일 많이 만난다.** 테스트에서 진짜 대신 끼우는 가짜의 총칭:

| 이름 | 뭘 하나 |
|---|---|
| **스텁** | 미리 정한 답만 돌려준다 — "결과가 맞나"를 묻는 테스트에 |
| **목(mock)** | 호출됐는지를 검증한다 — "이 함수를 불렀나"를 묻는 테스트에 |
| **페이크** | 진짜로 동작하는 간이 구현 (인메모리 DB) |
| **스파이** | 진짜를 부르되 기록도 남긴다 |

**3. 타입 스텁.** `.d.ts` — 구현 없이 타입만. `@types/node`가 이것.
**4. RPC 스텁.** 원격 호출을 로컬 함수처럼 보이게 하는 껍데기 (gRPC 클라이언트).

공통점: **껍데기가 진짜 자리에 앉아 있어서, 나머지를 먼저 짤 수 있다.** 문서 스텁도 같다.

## 이 프로젝트에서 — dummy.game.ts

단계 1의 "더미 문제"가 코드 스텁이었다. `dummy.game.ts`가 `LockRevealGame`을 구현하되
알맹이는 가짜로, 문제 은행 없이 리빌 루프를 완성했다. 아래층을 가짜로 막아야 위층을 진짜로
만들 수 있다 — "레이어로 짠다"를 물리적으로 가능하게 하는 장치다. 단계 2에서 진짜 게임
(quiz·betrayal)이 들어오며 더미는 삭제됐다 — **스텁은 그렇게 죽는 게 정상이다.** 아래
"눌러앉는 스텁"의 반대 사례.

스텁을 끼우려면 **주입 구멍**이 있어야 한다 — `loadRounds(ctx)`가 `ctx`(조·명단)를
주입받으므로 가짜로 돌릴 수 있다. (단계 2의 답: `LoadCtx`엔 끝내 DB 핸들을 안 넣었다.
퀴즈가 적재 함수만 주입받는 팩토리가 됐다 — `quiz.game.ts` "문제 은행(DB)은 코어 소유라
게임이 직접 못 잡는다". 주입 구멍이 DB째가 아니라 함수 하나로 좁혀진 것.)

### ★편한 스텁은 아무것도 안 잡는다★

스텁은 내가 만들어서 내 코드에 맞는다. 그게 함정이다 — 진짜 데이터가 오는 날 처음 깨진다.
더미가 계약 버그 2개를 찾아낸 건([[0001-계약을-먼저-정의하고-원칙을-타입에-박는다]]) 계약을
**진짜로 소비**했기 때문이다. 인라인으로 대충 채점했으면 전부 단계 2로 넘어갔다.

| 편한 스텁 | 아픈 스텁 |
|---|---|
| 짧은 문자열 | **제일 긴 것** — 화면 예산에 들어가나 |
| 예쁘게 떨어지는 개수 | **모자란 개수** — 덜 모였을 때 도나 |
| 정답이 하나 | **애매한 것** — 반쯤 맞으면? |
| 내가 정한 형식 | **진짜가 올 형식** — 아니면 나중에 갈아엎는다 |

**아픈 스텁은 스텁이 아니라 테스트다. 어차피 만들 거라 공짜다.**

## 언제 쓰고 언제 안 쓰나

- **순수 함수엔 스텁이 필요 없다.** 대체할 의존성(DB·시계·네트워크·랜덤)이 없으니 그냥
  부르면 된다. `score()`가 순수로 못 박힌 게 테스트 편의까지 같이 산 것이다 →
  [[순수-함수와-결정성]]. "스텁이 많이 필요하다"는 설계가 얽혀 있다는 신호다.
- **남용 지점 — 눌러앉는 스텁.** `not implemented`가 프로덕션에 나가는 건 실제로 흔하다.
  코드 스텁은 컴파일러를 속인다(통과하고 런타임에 터짐) — 문서 스텁이 밀려도 무해한 것과
  위험도가 다르다.
- **테스트를 전부 가짜로 감싸면** 진짜 코드는 아무도 안 돌린 게 된다. `verify.ts`는 반대로
  진짜 스키마에 진짜 조작 페이로드를 먹인다 — 그래서 그 OK가 뭔가를 보증한다.
