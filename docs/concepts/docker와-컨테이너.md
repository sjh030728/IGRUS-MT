# Docker와 컨테이너

**한 줄:** 프로그램을 실행에 필요한 것 전부와 함께 한 덩어리로 포장해서, 어느 컴퓨터에서든
똑같이 뜨게 하는 것. Postgres 같은 걸 "설치" 없이 띄울 때 제일 많이 쓴다.

## 왜 필요한가

[[postgres와-관계형-db]]를 쓰기로 했다고 치자. 그럼 노트북에 Postgres가 있어야 한다.
설치하면 되지 않나? 해보면 이렇게 된다:

- 설치 마법사가 뜬다. 버전을 고르라고 한다. 16? 15? 뭐가 다른지 모른다
- Windows 서비스로 등록된다. 부팅할 때마다 뜬다. 안 쓰는 날에도
- 포트 5432를 잡는다. 예전에 뭘 설치하다 5432를 이미 쓰고 있으면 조용히 실패한다
- superuser 비번을 정하라고 한다. 3주 뒤에 까먹는다
- MT 끝나고 지우려는데 깔끔하게 안 지워진다. 데이터 폴더가 남는다
- 그리고 **다음 행사 때 다른 노트북에서 다시 이 짓을 한다.** 그때 버전이 17이다

핵심 문제는 마지막이다. **재현이 안 된다.** "내 컴에선 되는데"가 여기서 나온다.
설치는 컴퓨터의 상태를 바꾸는 일이고, 그 상태는 아무 데도 안 적혀 있다.

## 컨테이너가 뭔가

**프로세스를 격리해서, 자기만의 파일시스템과 네트워크를 가진 것처럼 속이는 것.**

컨테이너 안의 Postgres는 자기가 깨끗한 리눅스에 혼자 있다고 믿는다. 실제로는 당신 노트북의
프로세스 하나일 뿐이다. 노트북에 설치되는 건 아무것도 없고, 컨테이너를 지우면 흔적이 0이다.

### 가상머신(VM)과 다른 점

VM은 **OS를 통째로** 하나 더 돌린다. 윈도우 안에 리눅스 전체가 뜬다. 무겁고(GB 단위)
느리다(부팅 1분).

컨테이너는 **호스트의 커널을 공유**하고 격리만 한다. 가볍고(MB 단위) 빠르다(1초).

(커널 = OS의 핵심 부분. 컨테이너가 리눅스 커널을 공유한다는 건, 윈도우에서 돌리려면
리눅스 커널이 어디선가 하나 필요하다는 뜻이다 — 이게 아래 위험의 원인이다.)

### 이미지 vs 컨테이너

**이미지 = 붕어빵 틀. 컨테이너 = 붕어빵.**

`postgres:16`이 이미지다. 누가 이미 만들어서 인터넷에 올려놨고, 받아서 찍어내면 된다.
Postgres 16이 정확히 그 버전으로, 설정까지 똑같이 뜬다. 100번 찍어도 100번 같다.

**이게 "재현이 안 된다"의 해답이다.** 설치 절차가 사람 머릿속이 아니라 이미지 안에 있다.

### Docker는 뭐고 컨테이너는 뭔가

컨테이너가 개념이고, Docker는 그걸 다루는 도구다. (요즘은 Podman 같은 대안도 있다.)
윈도우/맥에선 **Docker Desktop**이라는 앱을 깔아서 쓴다 — 리눅스 커널을 몰래 하나 띄워주는
게 그 앱의 주 업무다.

## 이 프로젝트에서

단계 1엔 Docker 파일이 하나도 없었다 — CLAUDE.md의 결정("로컬은 Docker")만 있고 파일이
없는 상태였다. DB가 단계 2였기 때문이다. 단계 2에서 레포 루트에 `docker-compose.yml`이
생겼다. 실물 요약:

```yaml
# docker-compose.yml (실물 요약 — 전문은 레포 루트)
services:
  db:
    image: postgres:16-alpine
    ports: ['5432:5432']
    environment:
      POSTGRES_PASSWORD: mt
    volumes:
      - mt-pgdata:/var/lib/postgresql/data
volumes:
  mt-pgdata:
```

`docker compose up` 한 줄이면 Postgres가 뜬다. 설치 없이, 다른 노트북에서도 똑같이.

읽는 법:

- **`image: postgres:16-alpine`** — 붕어빵 틀. `16`이 버전 태그다(`-alpine`은 더 작은 베이스
  이미지). `latest`를 쓰면 나중에 다른 게 받아져서 재현성이 깨진다. 숫자로 박는 게 맞다
- **`ports: ['5432:5432']`** — `호스트:컨테이너`. 컨테이너는 격리돼 있어서 기본적으로 밖에서
  못 들어간다. 이 줄이 "노트북의 5432로 오는 걸 컨테이너 5432로 넣어줘"라는 구멍 하나를 뚫는다
- **`volumes: mt-pgdata:/var/lib/postgresql/data`** — ★이 프로젝트에선 이 줄이 제일 중요하다★

### 볼륨을 빠뜨리면 원칙 4가 무너진다

컨테이너는 지우면 안이 통째로 사라진다. 그게 장점인데, **DB한테는 재앙이다.**

볼륨은 컨테이너 밖에 데이터를 두고 연결해주는 장치다. 이 줄이 없으면 컨테이너를 재시작하는
순간 `session_ledger`가 빈다. 그러면:

> **라운드 확정 점수는 매 라운드 스냅샷 저장하고 재시작 시 복구**한다.
> 로컬 프로세스가 죽어서 점수가 전부 날아가는 게 실제 시나리오다. (CLAUDE.md 원칙 4)

**점수를 지키려고 도입한 DB가, 설정 한 줄 때문에 점수를 날린다.** 컨테이너를 안 쓰면
없었을 위험이 컨테이너를 쓰면서 생긴 것이다.

실물은 안 빠뜨렸다 — 파일 주석이 직접 말한다: "데이터는 named volume에 남는다 —
컨테이너를 지워도 원장은 산다."

## 언제 쓰고 언제 안 쓰나

### 쓴다

- **DB, Redis 같은 미들웨어를 로컬에 띄울 때.** 이게 제일 흔하고 제일 이득이다.
  설치 안 하고, 버전이 박히고, 지우면 흔적이 0
- **팀이 여럿일 때.** "환경 세팅 문서" 대신 파일 하나
- **서버 배포.** 개발 환경과 운영 환경을 같게 만든다

### 안 쓴다 — 남용 지점

- **개발 중인 앱 자체를 컨테이너에 넣기.** 특히 윈도우에서. 호스트 파일을 컨테이너가
  들여다보는 구조라 파일 감시가 느려지고 hot reload가 죽는다. Node 앱은 그냥 `npm run dev`가 맞다
- **`latest` 태그.** 재현하려고 쓰는 도구인데 재현을 깬다
- **컨테이너 1개 띄우려고 Docker Desktop 깔기.** 아래 참고

### ★이 프로젝트에서 Docker가 진짜 위험한 지점★

CLAUDE.md는 "클라우드 배포 금지 / 사회자 노트북 로컬 실행"이다. Docker는 클라우드가 아니니
원칙 위반은 아니다. 하지만 **당일 운영 전제와 부딪히는 게 하나 있다.**

> 저녁 8시 시작, **8시 49분 종료**

7시 58분. 빔 연결하고, 공유기 켜고, QR 띄우고, 서버를 올린다. 그때 Docker Desktop이:

- 업데이트를 시작한다
- WSL2(윈도우가 리눅스 커널을 돌리는 장치)가 업데이트를 요구한다
- 재부팅을 요구한다
- 그냥 안 뜬다

**이건 실제로 흔하다.** 그리고 CLAUDE.md 위험목록에 없다. 폰 배터리는 있는데 이건 없다.

컨테이너의 이득("설치가 없다")은 **여러 컴퓨터에 반복 배포할 때** 나온다. 이 프로젝트는
노트북 한 대에 한 번 띄우고 3시간 뒤에 끝난다. 이득이 나오는 조건이 아니면서, 8시 정각에
새 실패 지점을 하나 얹는 것이다.

대안이 셋이었다:

1. **SQLite로 간다** — 파일 하나. DB 서버도, Docker도, 포트도 없다.
   [[postgres와-관계형-db]]의 "왜 Postgres냐" 참고
2. **Postgres를 그냥 설치한다** — 위의 설치 지옥을 한 번 겪지만, 당일엔 이미 서비스로 떠 있다
3. **Docker를 쓰되 운영으로 위험을 흡수한다** — 준비 절차와 리허설에 "띄워보기"를 넣는다

단계 2는 3안으로 갔고, 흡수 장치가 둘 붙었다: 서버가 부팅 때 DB 없으면 **딱딱하게 터지면서
할 일을 그대로 말하고**("docker compose up -d" — `db.service.ts`의 "부팅은 딱딱하게, 런타임은
부드럽게"), `program-ops.md` 시작 절차 첫 줄에 그 명령이 박혔다. "재부팅 후 처음부터
띄우기"를 리허설 체크리스트에 넣을지는 남은 판단이라, ADR 부재와 함께 findings 큐로
넘겨뒀다 → [[adr-아키텍처-결정-기록]]

**개념 노트로서의 교훈은 이거다:** Docker는 "설치의 재현성"을 사는 도구인데, 그 대가로
"Docker 자체가 떠야 한다"는 의존성을 산다. 재현할 일이 한 번뿐이면 대가만 남는다.
