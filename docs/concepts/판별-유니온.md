# 판별 유니온 (discriminated union)

**한 줄:** 공통 필드 하나(**판별자**)의 값에 따라 나머지 필드가 뭔지 확정되는 타입. 판별자를
확인하면 컴파일러가 나머지 모양을 알아낸다.

## 왜 필요한가

사회자 명령이 스무 개 남짓 있다. `ROUND_LOCK`, `VOID`, `SFX`, `SET_MULTIPLIER`, `SEED_SET`...
그런데 명령마다 필요한 데이터가 다르다. `VOID`는 "몇 번 기입을, 왜"가 필요하고,
`SFX`는 "무슨 소리"만 필요하고, `ROUND_LOCK`은 아무것도 필요 없다.

이걸 어떻게 타입으로 쓰나.

### 순진한 방법 1 — 명령마다 이벤트를 하나씩 만든다

```ts
socket.on('host:lock', ...)
socket.on('host:void', ...)
socket.on('host:sfx', ...)
// ... 나머지 전부
```

동작한다. 문제는 `packages/protocol/src/events.ts` 상단이 뭐라고 하는지다:

> `host:*` → 콘솔만. 서버가 토큰을 검사한다.
> (...) 폰이 host 이벤트를 쏠 수 있으면 그 순간 끝이다.

핸들러가 명령 수만큼 생기면 **인증 검사가 그만큼 흩어진다.** 감사 로그도, phase 합법성
검증도 전부 그만큼. 명령을 하나 더 추가하는 사람이 검사를 한 줄 빠뜨리면 그게 구멍이고,
컴파일러는 아무 말도 안 한다.

보안 경계는 좁을수록 좋다. 문이 스무 개면 스무 개를 다 잠가야 하고, 1개면 1개만 잠그면 된다.

→ 이벤트를 하나로 합친다. `host:cmd` 하나. **이제 payload 타입이 문제가 된다.**

### 순진한 방법 2 — optional 필드 자루

명령마다 필드가 다르니, 다 넣고 전부 optional(`?` = 없어도 됨)로 하면 되지 않나?

```ts
type HostCmd = {
  c: string;
  basePoints?: number;
  reason?: string;
  seq?: number;
  cue?: string;
  m?: 1 | 2;
  // ...
};
```

컴파일은 된다. 그런데 아래가 **전부 통과한다:**

```ts
handle({ c: 'VOID' });                       // 사유 없는 되감기. reason은 필수인데
handle({ c: 'SET_POINTS' });                 // basePoints 없음 → 배점이 undefined →
                                             //   점수가 NaN이 되어 원장에 박힌다
handle({ c: 'SFX', seq: 3, m: 2 });          // 뜻이 없는 조합
handle({ c: 'SET_POINS', basePoints: 100 }); // 오타. c가 string이라 아무거나 들어간다
```

그리고 서버를 짜면 이렇게 된다:

```ts
if (cmd.c === 'SET_POINTS') {
  cmd.basePoints;                 // number | undefined  ← 왜?? SET_POINTS면 있는 거잖아
  r.basePoints = cmd.basePoints;  // 컴파일 에러
}
```

`c`를 방금 확인했는데도 타입이 `undefined`를 못 뗀다. **"c가 'SET_POINTS'면 basePoints가 반드시
있다"는 사실을 타입이 모르기 때문이다.** 그래서 이미 아는 걸 매번 다시 검사하는 코드를 쓰게 된다.

문제의 본질: **어떤 필드가 존재하는지가 다른 필드의 값에 달려 있는데, 타입이 그 관계를 표현하지
못한다.** 판별 유니온은 정확히 이 관계를 표현하는 도구다.

## 판별 유니온

재료가 둘이다.

- **유니온:** `A | B` — 둘 중 하나. TypeScript 기본 문법이다.
- **리터럴 타입:** `'VOID'` — `string`이 아니라 **정확히 그 문자열 하나**만 되는 타입.
  Zod로는 `z.literal('VOID')`.

이 둘을 합친다. 각 멤버가 **서로 다른 리터럴**을 같은 이름의 필드에 넣는다.

```ts
export const HostCmd = z.discriminatedUnion('c', [
  z.object({ c: z.literal('ROUND_LOCK') }),
  z.object({ c: z.literal('SET_POINTS'), basePoints: z.number().int().positive() }),
  z.object({ c: z.literal('VOID'), seq: Seq, reason: z.string().min(1) }),
  z.object({ c: z.literal('SFX'), cue: SfxCue }),
  // ...
]);
```

`c`가 **판별자(discriminant)**다. `'ROUND_LOCK' | 'SET_POINTS' | 'VOID' | 'SFX' | ...`
중 하나이고, 그 값이 정해지면 나머지 필드가 자동으로 정해진다.

```ts
switch (cmd.c) {
  case 'SET_POINTS':
    cmd.basePoints; // number. optional 아님. 검사 안 해도 된다
    cmd.seq;        // ← 컴파일 에러: 이 타입엔 그런 필드가 없다
    break;
  case 'VOID':
    cmd.seq;        // Seq
    cmd.reason;     // string
    break;
  case 'ROUND_LOCK':
    cmd.basePoints; // ← 컴파일 에러. ROUND_LOCK엔 필드가 없다
    break;
}
```

이걸 **좁히기(narrowing)**라고 한다. 넓은 타입(명령 아무거나)이 코드 위치에 따라 좁은
타입(정확히 1개)으로 저절로 줄어드는 것. `if`, `switch`, `typeof` 같은 걸로 걸러낸
자리에서 컴파일러가 알아서 해준다.

오타 `c: 'SET_POINS'`는 어느 리터럴과도 안 맞으니 컴파일 에러다. 런타임에서도
`packages/protocol/verify.ts`가 확인한다:

```
[1] 사회자 명령 — 정상 통과 / 조작 거절
  OK   없는 명령 거절
  OK   ADJUST reason 없으면 거절 (아카이브가 이야기가 되게)
```

(타입이 있는데 왜 런타임 검사가 또 필요한지는 [[타입-vs-런타임-검증]].)

### 진짜 이득 — 명령을 하나 더 추가할 때

여기가 핵심이다. `default` 절에 `never`를 하나 두면:

```ts
function handle(cmd: HostCmd) {
  switch (cmd.c) {
    case 'SET_POINTS': return setPoints(cmd.basePoints);
    case 'VOID':       return voidEntry(cmd.seq, cmd.reason);
    // ... 나머지 전부
    default: {
      const _never: never = cmd;  // ★
      throw new Error('알 수 없는 명령');
    }
  }
}
```

`never`는 **값이 존재할 수 없는 타입**이다. 모든 case를 처리했다면 `default`에 도달하는
`cmd`는 남은 게 없으므로 타입이 `never`가 되고, 이 대입이 통과한다.

하나라도 빠뜨리면? `cmd`가 그 빠뜨린 멤버 타입으로 남아서 `never`에 대입이 **안 되고
컴파일 에러가 난다.**

즉 `HostCmd`에 멤버를 하나 더 추가하는 순간, **그 명령을 처리하지 않은 곳이 전부 빨간 줄로
뜬다.** 새 명령을 추가하고 서버 처리를 깜빡하는 게 불가능해진다. 이게 `events.ts` 주석이
말하는 것이다:

> 명령을 추가한다는 건 유니온 멤버를 추가하는 것이지 게이트웨이 핸들러를 늘리는 게 아니다.
> 여기가 보안 경계라 검사가 흩어지면 안 된다.

이 기법을 **빠짐없음 검사(exhaustiveness check)**라고 한다 → [[빠짐없음-검사와-never]]

**이 노트를 처음 쓸 땐 서버가 없었다.** 계약만 있었고 위 `handle`은 "앞으로 짤 코드"였다.
지금은 `apps/server/src/gateway/events.gateway.ts`에 그 `switch (cmd.c)`가 실제로 있다 —
**계약을 먼저 짜둔 덕에 서버가 저 안전망을 공짜로 받았다.** `0001`이 산 게 그거다.

## 이 프로젝트에서

계약 곳곳에서 쓴다. 판별자 이름이 `c`, `view`, `t`, `mode`, `kind`, `ok`로 제각각인 게 눈에
띌 텐데 — **이름은 아무래도 좋다.** 규칙은 "공통 필드 + 리터럴"뿐이다.

| 파일 | 타입 | 판별자 | 멤버가 뭔가 |
|---|---|---|---|
| `events.ts` | `HostCmd` | `c` | 사회자 명령 |
| `events.ts` | `PlaySnapshot` | `view` | 폰이 보는 화면 |
| `events.ts` | `Ack` / `SubmitAck` | `ok` | 성공 / 실패 |
| `display.ts` | `DisplayChunk` | `t` | 빔 정보 덩어리 |
| `display.ts` | `DisplayState` | `mode` | 빔 모드 |
| `ledger.ts` | `LedgerEntry` | `kind` | 원장 기입 |
| `live.ts` | `MatchOutcome` | `kind` | 라이브 매치 결과 |

> ★이 표는 개수를 안 적는다★ 전수는 `grep -rn 'z.discriminatedUnion(' packages/protocol/src/`가
> 답한다. 손으로 센 목록은 손으로 빠뜨린다 → [[하드코딩]], `0004`

이 중 셋이 서로 다른 것을 가르쳐준다.

### PlaySnapshot — 불가능한 화면을 만들 수 없게 한다

```ts
z.object({ view: z.literal('HEADS_UP'), me: Me, mine: z.unknown().nullable() }),
```

`HEADS_UP`(리빌 순간의 폰 화면)엔 `prompt`도 `roundId`도 `endsAt`도 **없다.** 주석:

> 리빌 순간 폰을 죽인다. 폰이 정답을 띄우면 40명이 고개를 박고 빔이 장식이 된다.

optional 자루였다면 `HEADS_UP`인데 실수로 `prompt`를 채워 보내도 타입이 통과한다. 판별
유니온이면 **그 필드가 타입에 존재하지 않아서** 채울 방법이 없다. 원칙이 코드 리뷰가 아니라
컴파일러에 걸린다 → [[0001-계약을-먼저-정의하고-원칙을-타입에-박는다]]

### DisplayState — "점수판 상시 노출"을 타입으로 만든다

```ts
z.object({ mode: z.literal('BLACK') }),                            // 점수판 없음
z.object({ mode: z.literal('AWARD'), scoreboard: Scoreboard }),    // 필수
z.object({ mode: z.literal('ROUND'), scoreboard: Scoreboard, round: DisplayRoundView }),
```

주석 그대로: "BLACK 말고 전부 scoreboard를 필수로 요구한다 → '점수판 상시 노출'이 타입이 된다."

`scoreboard?: Scoreboard` 하나로 뭉쳤으면 점수판 없는 `ROUND` 화면을 만들 수 있다. 멤버를
쪼갰기 때문에 **`BLACK`에서만** 점수판이 없을 수 있다. CLAUDE.md의 "타협 금지" 항목이
타협 불가능해진 것이다.

### LedgerEntry — 판별자에서 목록을 공짜로 뽑는다

```ts
export const VOIDABLE_KINDS = ['ROUND', 'ADJUST'] as const satisfies readonly LedgerEntry['kind'][];
```

`LedgerEntry['kind']`가 `'SEED' | 'ROUND' | 'ADJUST' | 'VOID'`다. 판별자 필드를 대괄호로
꺼내면 리터럴 유니온이 나온다 — **인덱스 접근 타입(indexed access type)**이라고 한다.
따로 enum을 하나 더 만들 필요가 없고, 만들면 어긋날 것이 하나 더 생긴다.

`satisfies`는 "이 배열의 원소가 저 4개 안에 드는지 검사하되, 타입은 `['ROUND','ADJUST']`
그대로 유지하라"는 뜻이다. (`: readonly ...[]`로 타입을 못 박으면 구체적인 값 정보가 날아간다.)

효과: `'ROUNDD'` 오타가 컴파일 에러. 나중에 `kind`에서 `'ADJUST'`를 지우면 여기서 걸린다.

공통 필드는 spread로 재사용한다:

```ts
const EntryBase = { seq: Seq, sessionId: SessionId, at: EpochMs, by: z.enum(['HOST','SYSTEM']) };

export const SeedEntry = z.object({ ...EntryBase, kind: z.literal('SEED'), totals: ... });
export const VoidEntry = z.object({ ...EntryBase, kind: z.literal('VOID'), voidsSeq: Seq, ... });

export const LedgerEntry = z.discriminatedUnion('kind', [SeedEntry, RoundEntry, AdjustEntry, VoidEntry]);
```

상속이 아니라 **복사**다. 각 멤버는 여전히 평평한 object 하나이고, 판별자만 각자 박는다.

## 언제 쓰고 언제 안 쓰나

### 쓴다

- 같은 자리에 오는 값이 여러 모양이고, **모양마다 딸린 데이터가 다를 때**
- `"이 필드는 X일 때만 의미 있음"`이라는 주석을 쓰고 있을 때. **그 주석이 신호다** —
  주석으로 쓰고 있다는 건 타입으로 쓸 수 있는 걸 안 쓰고 있다는 뜻이다
- 와이어를 건너는 것. 판별 유니온은 그냥 평범한 object라 JSON으로 그대로 나간다.
  클래스 상속으로 같은 걸 표현하면 `JSON.stringify` 한 번에 메서드가 다 날아간다

### 안 쓴다 — 남용 지점

**1. 멤버마다 딸린 데이터가 없으면 그냥 enum이다.** 이게 제일 흔한 과잉이다.

```ts
// 과잉
z.discriminatedUnion('t', [z.object({ t: z.literal('GOOD') }), z.object({ t: z.literal('BAD') })])

// 이 프로젝트가 실제로 한 것 — display.ts
export const Tone = z.enum(['NEUTRAL', 'GOOD', 'BAD', 'HOT']);
```

`Tone`, `SfxCue`, `RoundPhase`, `AutoTransition`은 전부 `z.enum`이다. 딸린 데이터가
없으니까. **판단 기준: 멤버마다 다른 필드가 있는가. 없으면 enum.**

(enum도 빠짐없음 검사가 된다. 그건 판별 유니온만의 이득이 아니다.)

**2. 판별자로 쓸 공통 필드가 없으면 못 쓴다.** `string | number`엔 공통 필드가 없다.
그냥 유니온이고 `typeof`로 좁힌다.

**3. 필드 차이가 하나뿐이면 nullable이 나을 수 있다.** 이 프로젝트에 실제 사례가 있다 —
`display.ts`의 `DisplayRoundView`:

```ts
/** ★COLLECT / COUNTDOWN 에서만 값이 있고 나머진 전부 null★ */
phaseEndsAt: EpochMs.nullable(),
```

이건 `phase`를 판별자로 쪼갤 수도 있었다. 그러면 "리빌 이후엔 타이머가 없다"가 주석이 아니라
타입이 된다. 안 쪼갠 이유: 멤버가 8개 나오는데 그중 7개가 서로 90% 똑같고, 빔 렌더러가
8갈래 분기를 하게 된다. 얻는 건 필드 하나의 정밀도다.

**이건 정답이 하나가 아닌 지점이다.** 판별 유니온은 멤버 간 차이가 클수록 이득이고,
차이가 필드 하나면 비용이 이득을 넘는다.

### 다른 언어에서 보면

같은 개념이 언어마다 이름이 다르다 — **sum type**, **tagged union**, **variant**.
Rust의 `enum`, Kotlin의 `sealed class`, Swift의 연관값 있는 `enum`이 전부 이거다.
그 언어들은 문법으로 직접 지원하는데, TypeScript는 전용 문법이 없어서 "리터럴 필드를
공통으로 넣는다"는 **관례**로 만든다. 그래서 판별자를 직접 골라 넣어야 한다.

TypeScript에서 "타입 여러 개 중 하나"를 표현할 때 클래스 상속이 아니라 이걸 쓴다.
