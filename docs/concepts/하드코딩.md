# 하드코딩 (hardcoding)

**한 줄:** 값을 코드나 문서에 직접 박아 넣는 것.
**문제는 박는 게 아니라, 이미 다른 데 있는 걸 또 박는 것이다.**

## 왜 필요한가 — 문서가 저절로 틀려졌다

`docs/README.md`에 이런 줄이 있었다:

```md
1. `concepts/타입-vs-런타임-검증.md` — 유일하게 채워진 노트. 나머지가 어떤 모양이 될지 보여준다
```

지금 채워진 노트는 6개가 넘는다. **언제 틀렸나? `/learn`을 한 번 돌린 순간이다.**
아무도 README를 건드리지 않았는데 문서가 틀려졌다.

`0004` 규칙 4("문서는 재고를 세지 않는다")가 이 사고에서 나왔다 —
**큐를 비우는 정상 동작이 문서를 깨뜨리는 구조**였던 것이다.

이게 하드코딩이다. "채워진 노트가 몇 개"라는 진실은 **`concepts/` 폴더 자체가 갖고 있다.**
README가 그걸 손으로 한 번 더 적었고, 폴더가 바뀔 때 README는 안 바뀌었다.

**하드코딩이 나쁜 이유는 "값을 적어서"가 아니다. 같은 진실이 두 곳에 살면 반드시
어긋나기 때문이다.** 그리고 어긋나도 아무것도 안 터진다 → [[조용한-실패]]

## 그럼 상수는 전부 하드코딩인가 — 아니다

`packages/protocol/src/display.ts`:

```ts
export const MAX_CONTENT_CHUNKS = 3;
export const ContentChunks = z.array(DisplayChunk).max(MAX_CONTENT_CHUNKS).readonly();
```

`3`이 박혀 있다. 하드코딩인가?

**아니다. 이 3은 다른 데 없다.** 위에 붙은 긴 주석이 그걸 보여준다 — 사람이 CLAUDE.md
"정보 덩어리 4개 이상 금지"를 읽고, 점수판이 하나를 먹고 시작한다는 걸 세고, 진행률 미터를
버리면 사망구간 해법이 사라진다는 걸 판단해서 나온 숫자다. **코드가 계산할 수 있는 게 아니다.**

**어디에도 없는 걸 처음 적는 건 하드코딩이 아니라 정의다.**

## 기준 — 이 값의 진실이 어디 있나

| 진실이 어디 | 어떻게 | 이 프로젝트 |
|---|---|---|
| **코드에 있다** | 파생한다 | `VOIDABLE_KINDS`의 `LedgerEntry['kind']` |
| **사람 판단에 있다** | 상수 + 이유 | `MAX_CONTENT_CHUNKS = 3` |
| **아직 모른다 (실측해야 함)** | 설정으로 뺀다 | `RateCap.perSec` |

**코드에 있으면 파생한다** — `ledger.ts`:

```ts
export const VOIDABLE_KINDS = ['ROUND', 'ADJUST'] as const satisfies readonly LedgerEntry['kind'][];
```

`'SEED' | 'ROUND' | 'ADJUST' | 'VOID'`를 손으로 또 안 적었다. `LedgerEntry['kind']`로
꺼냈다. `kind`를 하나 추가하면 자동으로 따라온다 → [[판별-유니온]]

**아직 모르면 값을 안 박는다** — `anticheat.ts`:

```ts
export const RateCap = z.object({
  perSec: z.number().int().positive(),
  burst: z.number().int().positive(),
});
```

**계약은 "상한이 존재한다"를 못 박고 "상한이 15다"는 안 박는다.** CLAUDE.md가
"인간 한계 **~**15tap/s"라고 하는데 그 물결표가 정직한 거다 — 진짜 값은 리허설에서
실측해야 나온다. 계약에 박아두면 리허설 결과를 반영하는 게 **계약 수정**이 된다.

## ★파생이 항상 답은 아니다 — 이 프로젝트의 실제 사고★

`0002`의 그 버그가 **하드코딩을 피하려다 난 것이다.**

```ts
z.object({ c: z.literal('DISPLAY_MODE'), mode: DisplayState.options[0].shape.mode }),
```

`'BLACK'`이라고 박기 싫었던 거다. `DisplayState`가 이미 모드를 아는데 또 적는 건
하드코딩이니까. 그래서 파생했다. **의도는 정확히 맞았다.**

결과: `options[0]`은 "어떤 모드든"이 아니라 "**첫 번째 멤버**"다. 나머지 3개 모드가
조용히 거절됐다. `0002`의 교훈:

> 유니온 **순서를 바꾸면 의미가 조용히 바뀌는** 코드였다.

차이가 뭐였나:

| 파생한 것 | 실제로 뽑은 것 | 안정적인가 |
|---|---|---|
| `LedgerEntry['kind']` | "kind가 될 수 있는 값 **전부**" | **의미다.** 멤버가 늘면 같이 늘어야 맞다 |
| `DisplayState.options[0]` | "**첫 번째로 적힌** 것" | **위치다.** 순서를 바꾸면 뜻이 바뀐다 |

**파생은 의미에서 하고 위치에서 하면 안 된다.** `[0]`, `.slice(1)`, "3번째 줄" 같은 게
나오면 그건 파생이 아니라 **우연에 기대는 것**이다.

(이 노트를 쓰는 세션에서 같은 실수를 또 했다. 스텁을 세려고 `awk 'FNR==3'`을 썼는데,
"3번 줄이 배너"라는 **위치** 규약이었다. 프론트매터 4줄이 붙자 조용히 `0`을 반환했다.)

**차라리 `z.literal('BLACK')`으로 박는 게 나았다.** 하드코딩이지만 순서에 안 흔들리고,
틀리면 시끄럽게 틀린다.

## ★테스트에서 하드코딩은 기능이다★

`packages/protocol/verify.ts`:

```ts
ok('3개 통과', ContentChunks.safeParse([chunk, chunk, chunk]).success);
ok('4개 거절', !ContentChunks.safeParse([chunk, chunk, chunk, chunk]).success);
```

`MAX_CONTENT_CHUNKS`를 안 쓴다. 3과 4를 손으로 박았다. **이게 맞다.**

만약 이렇게 썼다면:

```ts
Array(MAX_CONTENT_CHUNKS).fill(chunk)      // 통과해야 함
Array(MAX_CONTENT_CHUNKS + 1).fill(chunk)  // 거절돼야 함
```

누가 `MAX_CONTENT_CHUNKS`를 5로 바꿔도 **테스트가 그대로 통과한다.** 테스트가 대상에서
파생하면 대상이 바뀔 때 같이 바뀌어서 **아무것도 못 잡는다.**

**테스트는 "지금 코드가 뭘 하나"가 아니라 "뭘 해야 하나"를 적는 자리다.** 그 답은 코드 밖에
있고(CLAUDE.md), 그러니 손으로 적는 게 맞다. 중복 제거가 안 통하는 대표적인 자리다.

## 언제 쓰고 언제 안 쓰나

### 박는 게 맞다

- **진실이 코드 밖에 있을 때** — 사람 판단, 문서, 법. 단 **이름을 붙이고 이유를 적는다**
- **테스트** — 위 참고
- **파생이 위치에 기댈 때** — `options[0]`보다 `'BLACK'`이 낫다

### 박으면 틀린다

- 같은 진실이 이미 코드에 있을 때 → 파생 → [[파생-상태와-단일-진실-공급원]]
- **문서가 코드의 재고를 셀 때** → `0004`
- 환경마다 달라지는 것 (IP, 포트, 비밀번호) → 설정.
  이 프로젝트는 QR이 **로컬 IP**를 담는데, 숙소마다 다르다

### 매직 넘버

이름도 설명도 없이 혼자 있는 숫자다. `if (n > 100)` 같은 것.

```ts
n: z.number().int().positive().max(100),
```

`events.ts`의 이 `100`엔 이름이 없다. 대신 **바로 위에 주석이 있다** — "탭 1회당 1발이
아니라 100ms 모아서 보낸다. 폰이 개수만 세서 10Hz로 올린다."

**이름이든 주석이든, 6주 뒤에 왜 100인지 알 수 있으면 된다.** 나쁜 건 숫자가 아니라
**아무 설명 없이 혼자 있는 숫자**다. 상수로 빼는 건 수단이지 목적이 아니다.
