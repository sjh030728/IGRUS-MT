# 빠짐없음 검사와 never

> **스텁.** `/learn 빠짐없음 검사` 로 물어보면 채워진다. #스텁

**한 줄:** `never`(값이 존재할 수 없는 타입)에 대입을 시도해서, 유니온 멤버를 하나라도
처리 안 하면 컴파일 에러가 나게 만드는 기법.

개요는 [[판별-유니온]]의 "진짜 이득 — 명령을 하나 더 추가할 때"에 이미 있다.
여기는 그보다 깊은 것만 다룬다:

- `never` 타입 자체. `void`와 뭐가 다른가. 왜 "모든 타입의 부분집합"인가
- `assertNever(x: never): never` 헬퍼로 뽑아 쓰는 관용구
- `satisfies` 연산자 — `ledger.ts`의 `VOIDABLE_KINDS`가 쓴다. `as const`와 타입 표기(`:`)의 중간
- `z.enum`에서의 빠짐없음 검사. `RoundPhase` 8개, `SfxCue` 5개에도 똑같이 걸 수 있나
- `LEGAL_TRANSITIONS`가 `Record<RoundPhase, ...>`라서 이미 빠짐없음이 강제되는 것 —
  phase를 추가하면 전이표에 항목을 안 넣을 수가 없다. 같은 원리의 다른 얼굴

**궁금해할 만한 것:** `events.gateway.ts`의 `switch (cmd.c)`는 이 패턴을 쓰고 있나?
(힌트: 안 쓰면 명령을 하나 더 추가할 때 **컴파일러가 아무 말도 안 한다.** 스무 개 남짓
중에 하나를 빠뜨렸는지 누가 아나)
