# Postgres와 관계형 DB

**한 줄:** 프로그램이 죽어도 안 날아가는 곳에, 표(테이블) 모양으로 데이터를 넣어두고
SQL로 꺼내 쓰는 것. Postgres는 그런 DB 중 하나다.

## 왜 필요한가

8시 35분. 5번 문제까지 끝났고 3조가 1위로 올라와서 방이 뒤집어졌다. 그때 노트북이 죽는다.
배터리든, 실수로 누른 Ctrl+C든, 크래시든.

점수가 `let totals = {}` 같은 변수에만 있었으면 **전부 증발한다.** 40명이 보고 있는데
사회자가 "어... 잠깐만요, 다시 계산할게요"를 해야 한다. 그 순간 프로그램은 끝이다.

CLAUDE.md 원칙 4가 이 얘기다:

> 단, **라운드 확정 점수는 매 라운드 스냅샷 저장하고 재시작 시 복구**한다.
> 로컬 프로세스가 죽어서 점수가 전부 날아가는 게 실제 시나리오다.

"실제 시나리오다"라고 못 박아뒀다. 가능성이 아니라 예정이라는 뜻이다.

### 그럼 파일에 쓰면 되잖아

맞다. 그리고 이 프로젝트 규모에선 **거의 맞는 말이다.** 기입이 3시간에 50줄쯤 나오고,
쓰는 프로그램은 사회자 노트북의 서버 하나뿐이다. `fs.appendFile`로 한 줄씩 붙이면 된다.

그럼 DB가 뭘 더 해주나. 이 넷이다:

1. **동시에 써도 안 깨진다.** 두 군데서 같은 파일에 동시에 쓰면 줄이 섞인다. DB는 그걸 막는다.
2. **반쯤 쓰다 죽는 게 없다.** 파일에 쓰는 도중에 전원이 나가면 반 줄이 남는다. 그 파일은
   다음 실행 때 파싱이 터진다 — 복구하려고 만든 게 복구를 막는다. DB는 "다 됐거나 안 됐거나"
   둘 중 하나를 보장한다 (**트랜잭션** → [[트랜잭션과-acid]]).
3. **찾는 게 쉽다.** "3조의 무효 안 된 기입만" 을 파일로 하면 전부 읽어서 JS로 거른다.
   DB는 `WHERE`로 한 줄이다.
4. **모양을 강제한다.** 파일은 아무거나 쓸 수 있다. DB는 열이 정해져 있어서 `reason` 없는
   `VOID`가 애초에 안 들어간다.

2번이 이 프로젝트에서 제일 큰 이유다. **점수를 지키려고 만든 장치가 전원 차단 때 스스로
망가지면 안 된다.**

## 관계형 DB가 뭔가

**테이블 = 엑셀 시트 하나.** 열(column)이 정해져 있고, 행(row)이 데이터 하나다.
"관계형(relational)"은 테이블끼리 서로를 가리킬 수 있다는 뜻이다 (팀 테이블의 `id`를
점수 테이블이 참조하는 식).

**스키마 = 그 시트의 열 정의.** 열 이름과 타입. 미리 정해놓고 어기면 거절한다.
`packages/protocol`이 WS에서 하는 일을 DB가 저장소에서 하는 것이다.

**SQL = 테이블에 뭘 시키는 언어.** 특징은 **선언적**이라는 것 — "어떻게 찾아라"가 아니라
"뭘 원한다"만 쓰고, 어떻게 찾을지는 DB가 정한다.

```sql
SELECT * FROM session_ledger WHERE session_id = 's1' ORDER BY seq;
```

"session_ledger에서 s1 것만, seq 순서로 전부 줘." 반복문이 없다.

## 이 프로젝트에서

**★이 레포엔 DB 코드가 아직 한 줄도 없다★** 단계 0은 계약만 정의했다. 하지만 계약이
DB를 이미 세 군데서 전제하고 있다.

### 1. `ledger.ts` — 복구 절차가 이미 SQL로 적혀 있다

```
재시작 복구 = SELECT * FROM session_ledger ORDER BY seq → 위 fold → 끝.
```

테이블 이름(`session_ledger`)까지 정해져 있다. 이 한 줄이 원칙 4의 "재시작 시 복구" 전부다.
복구 코드를 따로 안 짜는 이유는 [[append-only-원장과-fold]]에 있다 — 쌓기만 하고 지우지
않으니, 복구가 "처음부터 다시 더하기"로 끝난다.

`LedgerEntry` 4종이 테이블로 내려가면 대충 이렇게 된다:

```
session_ledger
 seq | session_id | at            | by   | kind   | ...
   1 | s1         | 1721106000000 | HOST | SEED   | 낮 PG 점수
   2 | s1         | 1721106900000 | HOST | ROUND  | 퀴즈 1번
   3 | s1         | 1721107200000 | HOST | VOID   | 2번 무효
```

`kind` 열이 [[판별-유니온]]의 판별자 그대로다. **다만 판별 유니온을 테이블로 어떻게
내리냐가 아직 안 정해진 설계 논점이다** — 멤버마다 필드가 다르니, 열을 전부 펼치고 안 쓰는
건 NULL로 두든지(`SEED`엔 `voidsSeq`가 없다), 공통 열만 두고 나머지는 JSON 한 덩어리로
넣든지(Postgres의 `jsonb` 타입) 골라야 한다. 단계 2에서 부딪힐 문제다.

### 2. `game.ts` — 문제 은행이 Postgres에 있다

```ts
/**
 * 세그먼트 투입 시 1회. 문제은행(Postgres) 조회는 여기서 끝낸다.
 * ★라운드 진행 중 DB 접근 금지★ — 8시 35분에 쿼리가 느려지면 리빌이 늦는다.
 */
loadRounds(ctx: LoadCtx): Promise<readonly RoundSpec<TPlayPrompt>[]>;
```

DB를 **쓰는 시점을 계약이 제한한다.** 세그먼트 시작할 때 8문제를 통째로 읽고 끝. 라운드가
도는 동안엔 DB를 안 본다. 리빌이 늦으면 연출이 죽으니까 (원칙 1).

`docs/README.md`가 왜 문제 은행이 git이 아니라 DB에 있어야 하는지도 말한다 — 문항이
"김철수가 1학년 때 ~했다" 같은 실명 개인정보라서 공개 레포에 올리면 안 된다.

### 3. `events.ts` — 계약이 DB가 죽을 걸 이미 안다

```ts
health: z.object({
  db: z.enum(['OK', 'FAIL']),
  ...
}),
```

사회자 콘솔의 건강 패널에 `db` 항목이 있다. **DB가 죽어도 프로그램은 계속 돌아야 한다**는
뜻이다 — 실시간 상태는 메모리에 있으니 게임은 진행되고, 다만 사회자가 "지금 점수가 저장이
안 되고 있다"를 알아야 한다.

## 언제 쓰고 언제 안 쓰나

### 이 프로젝트가 DB를 **안** 쓰는 곳이 오히려 교훈이다

CLAUDE.md 원칙 4의 앞부분:

> **실시간 상태는 메모리, 확정 점수는 영속화**
> 3시간짜리 세션이다. 실시간 상태를 DB에 넣으면 동기화만 느려진다.

누가 뭘 제출했는지, 지금 phase가 뭔지, 탭이 몇 번 들어왔는지 — **전부 메모리다.**
DB에 안 넣는다. 초당 20번 바뀌는 걸 DB에 쓰면 DB가 병목이 되고 리빌이 늦는다.

| DB에 넣는다 | 메모리에 둔다 |
|---|---|
| 확정된 점수 기입 (`LedgerEntry`) | 진행 중인 phase |
| 문제 은행 | 제출 목록 |
| 끝난 뒤 아카이브 | 탭 카운터 |

**기준: 죽었을 때 다시 만들 수 있나.** 진행 중이던 라운드는 사회자가 20초면 다시 제시한다.
5번 문제까지의 점수는 아무도 다시 못 만든다. 그것만 DB로 간다.

`ledger.ts`가 이걸 직접 말한다:

> ★복구되는 것은 점수뿐이다★. 진행 중이던 라운드는 날아가고 사회자가 다시 제시한다(20초).
> 세그먼트 커서를 영속화하지 않는 이유: 사회자가 노트북을 손에 들고 서 있다.
> 어디였는지는 사람이 안다. 값어치 없는 복잡도다.

### 일반적인 남용 지점

- **세션/캐시를 관계형 DB에** — 만료되는 데이터는 Redis나 메모리가 맞다
- **설정을 DB에** — 파일이 맞다. git에 들어가고 diff가 보인다
- **로그를 DB에** — 파일이나 로그 수집기가 맞다
- **작은 앱에 Postgres** — 아래 참고

### 왜 Postgres냐 — 여기 논점이 하나 있다

관계형 DB는 여러 개다. Postgres, MySQL, **SQLite**...

앞의 셋과 SQLite가 다른 점: Postgres/MySQL은 **별도 프로그램**이라 따로 띄워놔야 하고
포트로 접속한다. SQLite는 **파일 하나**고 라이브러리로 끌어다 쓴다. 서버가 없다.

이 프로젝트의 조건을 보면 — 클라이언트가 1개(서버 프로세스뿐), 데이터가 3시간에 50줄,
배포 없음, 사회자 노트북 한 대. **이건 SQLite의 정확한 사용 조건이다.** 그리고 SQLite면
설치할 게 없어서 아래 [[docker와-컨테이너]]의 위험이 통째로 사라진다.

CLAUDE.md가 Postgres를 고른 이유는 명시돼 있지 않다. "이후 행사 재사용을 전제로 설계한다"와
`jsonb` 같은 기능 때문으로 읽히지만 추측이다.

**이건 개발 세션에서 판단할 것.** 개념 노트가 정할 문제가 아니다. 단계 2에서 원장 영속화를
짤 때가 결정 시점이고, 결정하면 `docs/decisions/`에 남는 게 맞다.
