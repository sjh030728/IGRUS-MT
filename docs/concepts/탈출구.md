# 탈출구 (escape hatch)

**한 줄:** 정상 경로가 막혔을 때 **사람이 손으로** 빠져나가는 길.
막힌 그 순간엔 못 만들기 때문에 미리 만들어둔다.

## 왜 필요한가 — 8시 35분엔 코드를 못 고친다

당연해 보이지만 여기서부터 시작해야 한다. **그 밤에 핫픽스는 불가능하다.**

- **49분짜리다.** 8시 시작, 8시 49분 종료
- 사회자는 **마이크를 들고 40명 앞에 서 있다.** 노트북 앞에 앉을 사람이 없다
- 배포도 롤백도 없다. `0001`이 런타임 assert를 반려한 이유가 이거다:
  > 8시 35분에 assert가 터지면 그건 방어가 아니라 **사고**다.

**그럼 뭔가 잘못되면 어떻게 하나?** 코드 말고 두 가지가 남는다.

**1. 재시작.** 원장이 이걸 전제로 설계됐다 → [[append-only-원장과-fold]]

> ★복구되는 것은 점수뿐이다★. 진행 중이던 라운드는 날아가고 사회자가 다시 제시한다(**20초**).
> (`ledger.ts`)

20초라고 적혀 있는 게 핵심이다. **재시작이 사고가 아니라 설계된 동작이다.**

**2. 콘솔.** 이게 탈출구다. CLAUDE.md의 사회자 콘솔 필수기능 목록:

> **현장은 100% 계획대로 안 간다. 이게 없으면 프로그램이 죽는다.**
> - 라운드 스킵 / 되감기
> - 점수 수동 보정
> - 예비 게임 즉시 투입

**"이게 없으면 프로그램이 죽는다"는 기능 얘기가 아니라 탈출구 얘기다.** 저 목록은 편의
기능이 아니라 **고칠 수 없는 49분을 견디는 장치**다.

## 이 프로젝트에서

| 무슨 일이 나면 | 탈출구 |
|---|---|
| 문제가 이상하다 (오타, 답이 둘) | `ROUND_ABORT` |
| 이미 점수가 들어갔다 | `VOID` — 되감기 |
| 점수가 틀렸다 · 번역 릴레이 채점 | `ADJUST` |
| 게임이 안 된다 | `ROUND_GOTO` · `SEGMENT_GOTO` — 스킵 |
| 시간이 남는다 | `SEGMENT_INJECT` — 예비 투입 |
| 시간이 없다 | `countdownMs`를 줄인다 (`phase.ts` 주석) |
| 카톡 알림이 빔에 떴다 | `DISPLAY_BLACKOUT` — 패닉 킬 |
| 폰 배터리가 죽었다 | `REISSUE_TOKEN` — 남의 폰으로 재접속 |
| 10초가 모자란다 | **★일부러 없다★** — 잠그고(SPACE) `NO_SUBMIT` 콜아웃으로 부른다 |
| 서버가 죽었다 | 재시작 → 원장이 fold |

### 상태머신 전체에 깔려 있다

`phase.ts`의 전이표를 세로로 읽으면 보인다:

```ts
IDLE:      ['PROMPT',    'ABORTED'],
PROMPT:    ['COLLECT',   'ABORTED'],
COLLECT:   ['LOCKED',    'ABORTED'],
LOCKED:    ['COUNTDOWN', 'ABORTED'],
COUNTDOWN: ['REVEAL',    'ABORTED'],
REVEAL:    ['REACTION',  'ABORTED'],
REACTION:  [],
ABORTED:   [],
```

**모든 phase가 `ABORTED`로 나갈 수 있다. 딱 하나 빼고.** `REACTION`은 커밋된 뒤라
나갈 곳이 없다 — 주석이 그렇게 적혀 있다: "커밋된 뒤엔 나갈 곳이 없다."

그럼 커밋 후엔 탈출구가 없나? 아니다. **`VOID`가 받는다.**

**탈출구가 끊기는 자리를 다른 탈출구가 이어받는다.** 커밋 전엔 `ABORT`(안 일어난 일로),
커밋 후엔 `VOID`(일어났지만 무효로). 둘의 차이가 곧 원장이 append-only인 이유다.

## ★탈출구는 들어가는 것보다 나오는 게 중요하다★

`0002`가 `DISPLAY_BLACK` **토글**을 반려한 이유:

> **나갈 문이 없다.** 패닉 킬은 왕복이다. 토글로 하면 되돌아오긴 하는데 멱등이 아니다 —
> 손이 떨려서 두 번 눌리거나 소켓이 재전송하면 카톡 알림이 빔에 **다시 뜬다.**
> 하필 그 순간이 절대 실패하면 안 되는 순간이다.

**탈출구는 급할 때 쓰는 물건이라 손이 떨린다.** 그때 한 번 더 눌러서 원래대로 돌아가는
설계는 탈출구가 아니다 → [[멱등성]]

그리고 반대편 안(모드를 직접 지정)을 반려한 이유가 더 깊다. 모드를 지정하는 방식에서
BLACK은 **목적지**다 — 알림 뜸 → BLACK → 처리함 → **이제 뭐였지?** 콘솔이 직전 모드를
기억해서 복원해야 하고, 그 복원이 틀리면 빔이 엉뚱한 화면으로 돌아온다. 오버라이드
방식에선 `on: false` 한 번이면 파생이 다시 보인다. **복원할 상태가 애초에 없다.**

**좋은 탈출구는 나올 때 복원할 게 없다.** 오버레이지 목적지가 아니다.
`blackout ? BLACK : 파생()` — 덮었다 걷으면 그 아래가 그대로 있다 →
[[파생-상태와-단일-진실-공급원]]

## 언제 쓰고 언제 안 쓰나

### 만든다

- **고칠 수 없는 시간대가 있을 때** — 무대, 라이브 방송, 배포 창, 마감
- **자동화가 판단할 수 없을 때** — 사람한테 손잡이를 준다

### 이게 원칙 3 위반이 아닌 이유

CLAUDE.md 원칙 3은 "앱이 사회자를 대체하면 안 된다"다. 탈출구는 **정반대**다 —
사회자한테 손잡이를 주는 것이고, **절대 자동으로 안 걸린다.**

- `MUTE_PARTICIPANT`: "★절대 자동으로 걸리지 않는다★ 사회자만 누른다"
- `anticheat.ts`: "앱은 절대 자동으로 점수를 깎지 않는다"

**자동 복구는 앱이 판단하는 것이고, 탈출구는 사람이 판단하는 것이다.** 같은 코드처럼
보여도 원칙 3의 반대편에 선다.

### 남용 지점 — 탈출구가 정상 경로가 되는 것

탈출구는 **드물게** 쓰여야 한다. 매번 쓰면 그건 탈출구가 아니라 설계 결함의 증상이다.
`ADJUST`를 매 라운드 누르고 있다면 채점이 틀린 거고, 고칠 곳은 `score()`다.

**"임시로 수동 처리"가 6개월 가는 건 이래서다.** 탈출구가 있으니 안 아프고, 안 아프니
안 고친다. 탈출구를 만들 땐 **얼마나 자주 쓰이는지 볼 방법**도 같이 만드는 게 좋다.

### 탈출구가 없으면 어떻게 되나

없는 게 아니라 **더 나쁜 게 생긴다.** 사회자가 그 자리에서 즉흥으로 만든다 —
빔을 손으로 끄거나, 노트북을 덮거나, "어... 잠깐만요"를 40명 앞에서 한다.
**탈출구를 설계하지 않는다는 건 탈출구를 없애는 게 아니라 최악의 탈출구를 고르는 것이다.**
