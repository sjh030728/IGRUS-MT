# 큐 (queue)

**한 줄:** 지금 못 하는 일을 쌓아뒀다가 나중에 꺼내 처리하는 목록. **꺼내면 사라진다**는
게 핵심이다.

## 왜 필요한가

학습 세션이 `DISPLAY_MODE` 버그를 찾았다. 그런데 못 고친다 — 권한도 없고, 무엇보다
고치는 방향이 둘인데 어느 쪽인지 판단할 설계 컨텍스트가 그 세션에 없다.

개발 세션은 고칠 수 있다. 그런데 **지금 안 열려 있다.**

**발견하는 시점과 처리하는 시점이 다르다.** 그 사이에 뭘 담아둘 데가 없으면 어떻게 되나?

채팅으로 "개발 세션에서 하시죠"라고 말한다 → 스크롤 위로 사라진다 → 다음 주에 개발 세션을
열면 그 세션은 이 버그를 모른다. 발견이 증발한다.

`docs/README.md`가 진단한 그 병이다:

> 채팅으로만 답하면 스크롤 위로 사라지고, 다음 달에 같은 질문을 받아 처음부터 다시
> 설명하게 된다. **그게 지금까지의 문제였다.**

**큐는 "지금 못 하는 일"을 잃어버리지 않게 하는 장치다.**

일반화하면: **만드는 쪽과 처리하는 쪽의 시점이나 속도가 다를 때 그 사이에 큐가 들어간다.**
만드는 쪽을 **생산자(producer)**, 처리하는 쪽을 **소비자(consumer)**라고 한다. 큐가 없으면
생산자는 소비자가 받을 때까지 기다려야 하고, 그러면 둘이 한 몸으로 묶인다.

## 자료구조로서의 큐

원래는 자료구조 이름이다.

**FIFO** (First In, First Out) — 먼저 들어온 게 먼저 나간다. 매표소 줄이다.

- **enqueue** — 뒤에 넣는다
- **dequeue** — 앞에서 꺼낸다

반대가 **스택(stack)** — LIFO. 접시 쌓기다. 나중에 올린 게 먼저 내려온다.

JS 배열로 보면 차이가 한 글자다:

```js
const q = [];
q.push(1); q.push(2);
q.shift();              // 앞에서 꺼낸다 → 1.  큐

const s = [];
s.push(1); s.push(2);
s.pop();                // 뒤에서 꺼낸다 → 2.  스택
```

## ★이 프로젝트의 "큐"는 사실 큐가 아니다★

여기가 이 노트에서 제일 중요하다.

큐의 정의적 성질이 **둘**이다:

1. **순서** — FIFO
2. **소비되면 사라진다** — drain

**이 프로젝트는 2번만 빌려 쓰고, 1번은 명시적으로 거부한다.**

`findings.md`가 2번을 말한다:

> 개발 세션이 가져가서 처리하고 **지운다** (체크만 하지 말 것 — **큐지 로그가 아니다**).

`docs/README.md`가 1번을 거부한다:

> **순서는 여전히 없다.** 영역은 "이게 뭐에 관한 건가"만 말하고 "뭘 먼저"는 말하지 않는다.

왜 거부했나. 바로 위에 이유가 적혀 있다:

> **게이트는 없다. 영원히.** "이거 배우고 넘어가세요"를 만들면 학습이 숙제가 되고,
> 숙제가 되면 죽는다.

스텁이 FIFO였으면 꽂힌 순서대로 배워야 한다. 그러면 지금 궁금하지도 않은 걸 먼저 배우게
되고, 그게 숙제다. **순서를 넣는 순간 이 시스템이 죽는다.**

**그래서 엄밀히 말하면 이건 큐가 아니라 백로그(backlog)다** — 순서 없는 미처리 작업 집합.
그런데도 "큐"라고 부르는 건 **비워야 한다**는 쪽을 강조하려는 것이다. 이 시스템이 지키려는
게 순서가 아니라 배수(drain)라서.

**용어를 느슨하게 빌려 쓴 것이고 그게 의도다.** 다만 면접에서 "큐가 뭐냐"고 물으면
FIFO라고 답해야 한다. 여기서 쓰는 뜻이 표준이 아니다.

## 큐 vs 로그 — 이 프로젝트에 둘 다 있다

| | 큐 | 로그 |
|---|---|---|
| 처리하면 | **사라진다** | **남는다** |
| 목적 | 미래를 안 잊는 것 | 과거를 설명하는 것 |
| 여기선 | `findings.md`, `#스텁` 태그 | `session_ledger`, `decisions/` |

`ledger.ts`의 점수 원장은 **로그**다 — "UPDATE도 DELETE도 없다." 되감기조차 지우는 게
아니라 VOID 기입을 하나 더 추가한다 → [[append-only-원장과-fold]]

ADR도 로그다 — 틀린 판단도 안 지우고 상태만 `폐기됨`으로 바꾼다 → [[adr-아키텍처-결정-기록]]

`findings.md`는 큐다:

> 처리한 항목을 체크만 하고 남겨두면 파일이 길어지고, 길어지면 안 읽고, 안 읽으면 큐가 죽는다.

**왜 정반대냐:** 로그는 과거를 설명하는 게 목적이라 지우면 손해다. 큐는 "남은 게 뭔지"를
한눈에 보여주는 게 목적이라 안 지우면 못 읽는다. **목적이 반대라 보존 정책도 반대다.**

같은 파일에 둘을 섞으면 둘 다 죽는다. 그래서 findings 항목 중 남길 가치가 있는 건
항목으로 남기는 게 아니라 `decisions/`로 **승격**시킨다. 큐에서 빼서 로그에 넣는 것이다.

## 스텁 큐는 어디 있나 — 파일이 아니라 태그다

스텁은 채워도 파일이 안 지워진다. 그럼 큐가 아닌가?

**큐의 원소가 파일이 아니라 태그다.** `#스텁`이 붙은 것들의 집합이 큐고, 태그를 떼면
큐에서 나간다. 파일은 자산으로 남는다 → [[스텁]]

> `/learn`이 노트를 채우면 태그를 뗀다. **그 숫자가 줄어드는 게 진행률이다**

findings와 다른 점: findings 항목은 처리하고 나면 남길 게 없다. 스텁은 채워지면 그 자체가
자산이다. **그래서 하나는 파일을 지우고 하나는 태그를 뗀다.** 큐에서 나가는 방식이 다를 뿐
"처리되면 큐에서 사라진다"는 같다.

## 코드에서 — 큐를 쓸 곳과 쓰면 안 되는 곳

### ★쓰면 안 되는 곳★ — `live:frame`

```ts
/** ★display room만. 20Hz. volatile★ 스냅샷 투영기를 우회하는 유일한 채널이다. */
'live:frame': (f: LiveFrame) => void;
```

**`volatile`이 "큐에 쌓지 마라"는 뜻이다.** Socket.IO 옵션인데, 클라이언트가 지금 못 받는
상태면 그 메시지를 **그냥 버린다.** 기본 동작(쌓아뒀다 재전송)의 반대다.

왜 버리냐. 20Hz 프레임은 **신선도가 전부**다. 빔이 잠깐 버벅여서 프레임 40개가 밀렸다고
치자. 큐라면 그걸 다 보낸다 → 빔이 2초 전 탭 줄다리기를 슬로우모션으로 재생한다.
폰은 이미 끝난 매치를 두드리고 있는데.

**늦게 온 데이터가 아예 없는 것보다 나쁘면 큐를 쓰면 안 된다.** 최신 것 하나만 있으면
되니까 버리는 게 맞다.

### 큐처럼 보이는데 큐가 아닌 것 — `PlayTap`

```ts
/**
 * 탭 배치. ★탭 1회당 1발이 아니라 100ms 모아서 보낸다★
 * 폰이 개수만 세서 10Hz로 올린다. 배터리(위험 3위)와 대역폭 둘 다에 필요하다.
 */
export const PlayTap = z.object({ matchId: MatchId, n: ..., windowMs: ... });
```

손가락은 초당 15번 두드리는데 전송은 10Hz다. 속도가 다르니 사이에 뭔가 필요하다 —
여기까지는 큐의 교과서적 상황이다.

**그런데 큐를 안 썼다.** `n: number` — **개수만 센다.** 탭 12개를 배열로 들고 있다가
보내지 않는다.

이유: **탭은 서로 구별되지 않는다.** 12번의 탭이 각각 뭐였는지가 아무 의미도 없고 개수만
의미가 있다. 구별이 안 되면 큐가 아니라 **카운터**로 충분하고, 카운터가 훨씬 싸다.

**"모아뒀다 처리한다"고 전부 큐가 아니다.** 원소를 하나씩 꺼내 각각 처리해야 큐고,
합쳐서 숫자 하나가 되면 그건 집계다.

## 언제 쓰고 언제 안 쓰나

### 쓴다

- 생산과 소비의 **시점**이 다를 때 — `findings.md`(학습 세션 ↔ 개발 세션)
- 생산과 소비의 **속도**가 다를 때 — 작업 큐, 메시지 큐
- **순서가 의미 있을 때**

### 안 쓴다

| 상황 | 대신 | 이 프로젝트의 예 |
|---|---|---|
| 늦은 게 없는 것보다 나쁘다 | 버린다 | `live:frame` volatile |
| 원소가 서로 구별 안 된다 | 카운터 | `PlayTap`의 `n` |
| 순서를 강요하면 해롭다 | 백로그 (순서 없음) | `#스텁` 태그 |

### 남용 지점 — 안 비는 큐

**큐가 안 비면 큐가 아니다.** 쌓이기만 하는 큐는 로그고, 그것도 아무도 안 읽는 로그다.
"나중에 볼 것" 폴더가 그렇게 죽는다.

이 프로젝트가 이걸 강박적으로 경계한다. `findings.md`의 **정상 상태가 비어 있는 것**이고,
파일에 대놓고 그렇게 적혀 있다:

> *비어 있음. 이게 정상 상태다 — 큐는 쌓이라고 있는 게 아니라 비우라고 있다.*

**큐를 만들었으면 비우는 쪽이 누군지를 같이 정해야 한다.** 안 정하면 그건 큐가 아니라
그냥 쌓이는 파일이다. README가 큐마다 "쌓는 쪽 / 비우는 쪽"을 표로 못 박은 게 그래서다.
