## 시간 산정 원칙

**리액션 시간을 실행 시간과 별도로 잡는다.** 다들 이걸 빼먹어서 항상 어긋난다.
내부 퀴즈는 **실행 8분에 리액션 8분**이다 (= 위 표의 16분. 문제당 실행 60초 / 리액션 60초).
정답 공개하고 "OO이 나와봐" 하는 그 시간이
콘텐츠이지 낭비가 아니다. 앱이 점수만 띄우고 넘어가면 게임의 절반을 버리는 것.

## 진행 판단

- **조 배정에 학번을 섞는다.** 내부 퀴즈는 신입생이 못 맞힌다. 고학번 몰린 조가 유리해짐.
  신입생 관련 문제도 반드시 끼워서 소외를 막는다.
- **설문 항목은 임원이 사전 필터링한다.** 그냥 받으면 "나는 라면을 좋아한다" 같은 게 온다.
  설문에 강한 예시를 주고, 재미없는 건 뺀다. 안 하면 그 구간이 통째로 죽는다.
  → **문안·필터 기준·목표 응답 수는 `docs/survey.md`에 있다.** 복붙해서 바로 보내면 된다.
  (원래 두진실 항목에 대한 지침이었는데 두진실이 1인 개발로 빠졌다. 다만 이 판단은
  내부 퀴즈 문항에 그대로 적용된다 — 오히려 퀴즈가 유일한 설문 의존 콘텐츠가 되면서 더 중요해졌다.)
- **배신 라운드로 끝나도 괜찮다.** 원래 뒷맛이 문제인데, 바로 술자리로 이어지면
  "야 니네 왜 배신했어"가 최고의 안주가 된다. 앙금이 대화 소재로 전환됨.
  단 시상 + 단체사진은 반드시 붙여서 감정 정리.

## 상품

술자리와 연결하되 술 자체와는 연결하지 않는다.
- 좋음: 자리 선점권, 안주 선택권, 첫 건배사 지정권
- 안 됨: 벌칙 지정권, 면제권 — 음주량과 엮이면 안 마시는 사람이 소외되고 강요 구조가 된다

## 서버가 죽으면 (당황 금지 — 설계된 동작이다)

1. **재시작만 한다.** 확정 점수는 DB에 있어서 점수판이 그대로 이어진다. 빔·폰은 자동 재접속.
2. **점수판을 본다.** 맞으면 끝. 진행 중이던 문제 하나만 날아갔다 — 다시 제시하면 된다 (20초).
3. **이상하면 낮 점수를 콘솔에서 다시 입력한다.** 두 번 넣어도 두 배가 되지 않는다 (SEED는 덮어쓰기).

시작 전 준비는 두 줄: `docker compose up -d` → 서버 실행. DB가 안 떠 있으면 서버가
부팅을 거부하며 저 명령을 알려준다. 문항 넣기는 `npm run db:seed -w @mt/server`.

당일 콘솔 토큰: `session.config.json`의 `hostToken`을 채우고 콘솔은 `/host?token=<그 값>`으로
연다. 안 채우면 개발 기본값(mt-host)이라 ★참가자도 콘솔을 열 수 있다★ — IT 동아리다.

폰이 죽은 사람: 콘솔 사람 패널에서 그 사람의 **코드** 버튼 → 응답 패널의 4글자를 불러준다 →
빌린 폰의 "코드 입장"에 입력. 원래 폰은 자동으로 밀려난다.

## 리허설 (단계 5)

개발자가 혼자 폰 2개로 테스트하는 건 리허설이 아니다. **동아리원 10명 실제 소집.**
공유기 세팅까지 포함해서 동방에서 실제 조건으로 돌린다.

타임라인 그대로 재보면 대부분 초과한다. **그때 뭘 자를지 미리 정해둘 것.**

- [ ] **탭 줄다리기 실기기 6대 완주** — 단계 3의 E2E는 소켓 시뮬레이션이다. 실제 폰의
  터치 연타 속도로 K(민감도)·DURATION(30초)·RATE(15/25)를 확정한다 (`tap-tug.game.ts`
  상수 3개). KO가 안 나오면 K를 올리고, 10초 만에 끝나면 내린다.
- [ ] **노트북 재부팅 → 처음부터 띄우기** — Docker Desktop 자동 시작이 꺼져 있으면 당일
  재부팅 한 번에 DB가 없다. `docker compose up -d` → 서버 → 점수판 복구까지 눈으로 확인.
