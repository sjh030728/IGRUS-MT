# 0006. 원장의 진실은 메모리, DB는 미러 — 부팅은 딱딱하게, 런타임은 부드럽게

- 날짜: 2026-07-16 (단계 2)
- 개념: [[쓰기-지연-미러]], [[큐]], [[append-only-원장과-fold]]

## 무슨 결정인가

원장 기입은 메모리에 먼저 쌓이고(즉시, 동기) DB엔 큐를 통해 비동기로 미러된다.
seq도 메모리가 매긴다. 재시작 복구는 부팅 때 `session_ledger`를 읽어 배열만 되살린다 —
fold(`totals`)는 자기가 뭘 읽는지 모른 채 그대로 돈다.

DB 생사는 시점에 따라 다르게 취급한다:

| | DB가 죽어 있으면 | 왜 |
|---|---|---|
| **부팅 (7:30)** | ★서버가 안 뜬다★ — `docker compose up -d`를 에러로 알려준다 | 사람이 고칠 수 있는 시간이다. config 오타와 같은 급 |
| **런타임 (8:35)** | 쇼는 계속 — 미러 큐가 쌓이며 3초마다 재시도, 콘솔 health.db만 FAIL | 무대를 세울 수는 없다. 점수의 진실은 메모리라 게임엔 지장이 없다 |

`INSERT … ON CONFLICT DO NOTHING`이라 재시도가 겹쳐도 두 번 안 쌓인다 ([[멱등성]]).

## 왜

- **커밋(리빌 직후)이 DB 지연에 볼모로 잡히면 안 된다.** REVEAL → REACTION 사이에
  쿼리가 낀 구조면, 8시 35분에 디스크가 버벅이는 순간 리빌이 늦는다 — 성공 기준 직격.
- 원칙 4의 문장 그대로다: "실시간 상태는 메모리, 확정 점수는 영속화." 영속화는
  ★쓰기 경로의 성질★이지 읽기 경로가 아니다. 읽기(점수판)는 매 스냅샷 도는 뜨거운 경로다.

## 안 고른 것

- **DB가 진실 (write-through, seq는 시퀀스가 발급)** — 커밋마다 왕복 대기. 로컬 도커라
  평소엔 1ms지만, 죽는 순간 커밋 버튼이 먹통이 된다. 점수를 지키려던 장치가 진행을 죽인다.
- **런타임도 딱딱하게 (DB 죽으면 커밋 거부)** — 배신·탭 줄다리기는 DB가 없어도 되는
  게임이다. 미러 하나 때문에 전 프로그램을 세우는 건 배보다 배꼽.
- **부팅도 부드럽게 (DB 없이도 뜸)** — 퀴즈가 조용히 빈다. 시끄러운 7:30이 조용한 8:03보다 낫다.

## 이 결정이 사는 곳

- `apps/server/src/core/db.service.ts` — 헤더 주석 · 미러 큐 · `drain()`
- `core/ledger.service.ts` — `onModuleInit`(복구) · `append`(메모리 우선)
- `docs/program-ops.md` "서버가 죽으면" — 이 설계의 사용자 매뉴얼 3줄
