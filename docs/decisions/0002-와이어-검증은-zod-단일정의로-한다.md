# 0002. 와이어 검증은 Zod 단일 정의로 한다

- 날짜: 2026-07-15
- 상태: 확정
- 개념: [[타입-vs-런타임-검증]], [[스키마-기반-타입-추론]], [[판별-유니온]]

## 무슨 결정인가

WS를 건너다니는 모든 데이터를 Zod 스키마로 한 번만 정의하고, TypeScript 타입은 `z.infer`로
뽑아 쓴다. 타입 선언과 검증 함수를 따로 두지 않는다.

## 왜

**CLAUDE.md 위험목록 1위가 치팅이고, 그게 이 결정을 강제한다.**

> "IT동아리다. 탭 줄다리기에서 콘솔로 이벤트 뿌리는 시도가 반드시 나온다."

**반드시**라고 적혀 있다. 가능성이 아니라 예정이다. 그러면 런타임 검증은 선택이 아니다.

그런데 TypeScript 타입은 컴파일 후 사라지므로 런타임에 아무것도 못 막는다. 그래서 보통
정의를 두 벌 쓰게 되고(타입 하나, 검증 함수 하나), 둘이 어긋나는 게 시간문제다. Zod는
한 벌로 둘 다 준다.

자세한 건 [[타입-vs-런타임-검증]].

## 안 고른 것

**순수 TS 타입만** — 가장 가볍지만 폰에서 오는 데이터를 못 막는다. 위험목록 1위와 정면 충돌.

**Nest class-validator DTO** — Nest 관용이라 서버만 보면 자연스럽다. 하지만 데코레이터 기반이라
브라우저(빔/폰)에서 재사용이 어렵고, 그러면 빔·폰이 타입을 따로 들게 된다. **계약이 두 벌 나는 것**
— 0001을 무효화한다.

## 이 결정이 사는 곳

- `packages/protocol/src/events.ts` — WS 이벤트 스키마 전부
- `packages/protocol/src/game.ts` — **예외.** 여기만 순수 TS다. 이유는 파일 상단 주석에 있다.
  와이어를 안 건너는 건 Zod를 쓸 이유가 없다. 검증이 사는 유일한 경계는 `parseAnswer()`의
  `unknown → ParseResult<T>` 지점
- `packages/protocol/verify.ts` §1, §2 — `n: 999999`짜리 가짜 탭이 실제로 거절되는지
