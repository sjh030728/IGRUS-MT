# 0006. 학습 세션은 읽기 전용으로 두고 findings로 넘긴다

- 날짜: 2026-07-16
- 상태: 확정
- 개념: 없음 — 이건 개발 지식이 아니라 작업 방식 결정이다

## 무슨 결정인가

학습 세션(`/learn`)에 코드 수정 권한을 주는 안을 검토했고 **뺐다.**
대신 `docs/findings.md`를 만들었다 — 학습 세션이 발견을 append 하고 개발 세션이 처리하고 지운다.

`concepts/` 스텁과 **정확히 대칭**이다. 방향만 반대다.

| 큐 | 쌓는 쪽 | 비우는 쪽 |
|---|---|---|
| `concepts/` 스텁 | 개발 세션 | 학습 세션 |
| `findings.md` | 학습 세션 | 개발 세션 |

## 왜

**불편은 진짜였다.** 학습 세션이 버그를 발견하면 "개발 세션에서 하시죠"라고 말하고 끝난다.
사용자가 터미널을 옮겨서 다시 설명해야 한다. 두 번 일하는 것처럼 보였다.

**근거가 된 건 [[0005-빔-모드는-세그먼트에서-파생하고-콘솔은-덮기만-한다]] 그 자체다.**
학습 세션(`/learn 판별 유니온`)이 `DISPLAY_MODE` 버그를 정확히 찾아냈다. 발견은 완벽했다.
그런데 **고칠 수는 없었다:**

- 수정 방향이 (a)/(b) 정반대 둘이었고, 어느 쪽인지는 코드를 더 읽어서 알 수 있는 게
  아니었다. 서버가 아직 없어서 `SEGMENT_GOTO`가 모드를 어떻게 다루는지가 **존재하지 않았다.**
  당일 라인업과 세그먼트의 관계를 알아야 갈리는 문제였고, 그건 CLAUDE.md 제품 판단이지
  타입 지식이 아니다.
- 권한이 있었으면 **(a)로 고쳤을 것이다.** 버그 리포트가 가장 자연스럽게 시사하는 수정이고,
  증상만 보면 맞다. 그리고 0005가 기록했듯 **틀린 수정이었다.**
- 하필 대상이 계약이다. 계약은 되돌리기가 제일 비싼 자리다([[0001-계약을-코드보다-먼저-정의한다]]).
- 그 순간 사용자는 타입 이론 설명을 읽는 중이었다. **diff를 리뷰할 수 있는 상태가 아니다.**
  리뷰 없는 수정이 계약에 들어가는 건 1인 개발에서 제일 비싼 사고다(0001: "혼자면 리뷰해줄
  사람이 없어서 계약이 어긋난 걸 훨씬 늦게 안다").

**핵심은 진단이 틀렸다는 것이다.** "두 번 일한다"의 원인은 권한이 아니라
**발견이 채팅에서 증발하는 것**이었다. 스크롤 위로 사라지니까 다시 설명하게 된다.
findings.md가 그걸 고친다 — 권한을 주는 건 없는 병에 약을 쓰는 것이고, 부작용은 계약 오염이다.

**발견은 싸고 결정은 비싸다.** 학습 세션은 싼 쪽을 아주 잘한다 — 코드를 정독하니까.
비싼 쪽은 컨텍스트가 없다. 잘하는 것만 시키고 넘긴다.

## 안 고른 것

**학습 세션에 수정 권한 부여** — 위가 전부 이유다. 검토했고, 실제 사례에서 틀린 수정을
냈을 거라는 게 확인돼서 뺐다.

**"안전한 것만" 수정 허용 (오타, 주석 같은 것)** — 경계가 안 그어진다. `DISPLAY_MODE` 건도
`options[0]` → `options` 한 글자 수정처럼 보였다. **한 글자가 계약 변경이었다.**
diff 크기는 위험도의 지표가 아니다.

**채팅으로 넘기고 사용자가 개발 세션에 복붙** — 지금까지의 방식이고, 이게 병이다.
사용자가 기억해서 옮겨야 하면 그 트랙은 죽는다 (CLAUDE.md: 설문이 죽는 것과 같은 이유).

**GitHub 이슈** — 인터넷 의존이고, 이 프로젝트의 기록은 전부 레포 안 파일이다(옵시디언 보관함이
`docs/`다). 큐만 다른 데 있으면 안 본다.

## 처리한 항목은 지운다 — 체크가 아니라 삭제

스텁과 반대다. 스텁은 채워지면 **남는다**(자산이니까). findings 항목은 처리되면 **지운다.**

로그가 아니라 큐다. 처리 완료 항목이 쌓이면 파일이 길어지고, 길어지면 안 읽고,
안 읽으면 큐가 죽는다. 남길 가치가 있는 건 항목이 아니라 **결정**이고, 그건 `decisions/`로
승격된다 — 이 ADR과 0005가 그 승격의 첫 사례다.

## 이 결정이 사는 곳

- `docs/findings.md` — 큐 자체
- `.claude/commands/learn.md` 규칙 1 — 학습 세션이 append 하는 법. 방향이 갈리면
  **고르지 말고 둘 다 쓰라**는 게 핵심이다
- `docs/README.md` "findings가 반대 방향 큐다" — 두 큐의 대칭

## 왜 규칙이 세 곳에 있나

한 곳에 있으면 안 읽힌다. 각각이 **읽히는 순간**이 다르다 —
`learn.md`는 학습 세션이 발견한 순간, `README.md`는 사용자가 시스템을 이해할 때,
`findings.md`는 개발 세션이 큐를 비울 때. 전문은 `learn.md`에 있고 나머지는 링크다.

한동안 `findings.md` 자신이 규칙의 유일한 사본이었다. **항목을 지우다가 파일째 날리면
규칙도 같이 사라지는 상태였다.** 큐와 큐의 규칙이 같은 파일에 살면 안 된다.
