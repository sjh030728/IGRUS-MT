# 0001. 계약을 코드보다 먼저 정의한다

- 날짜: 2026-07-15
- 상태: 확정
- 개념: [[스키마-기반-타입-추론]], [[모노레포와-워크스페이스]]

## 무슨 결정인가

단계 0에서 코드를 거의 안 쓴다. 대신 서버·빔·폰·콘솔이 주고받을 메시지의 **모양**을
`packages/protocol/`에 TypeScript로 못 박는다. 로직은 0줄이다.

## 왜

**메시지가 4개 주체가 만나는 유일한 지점이기 때문이다.** 서버가 점수를 `score`로 보내고
빔이 `points`로 기대하면, 그 불일치는 코드를 짜는 동안엔 안 보이다가 합치는 날 터진다.
계약을 코드로 박아두면 컴파일러가 그날이 아니라 그 자리에서 잡는다.

2인 시절엔 이게 분담(A=서버·콘솔 / B=빔·폰) 때문에 필수였다. **1인이 되면서 오히려 더
중요해졌다** — 혼자면 리뷰해줄 사람이 없어서 계약이 어긋난 걸 훨씬 늦게 안다.

그리고 계약이 제자리에 있으면 게임 하나가 30~50줄로 끝난다. 200줄이면 계약이 잘못된 것이고,
그걸 **게임 코드 길이로 측정할 수 있다**는 게 이 결정의 부수 효과다.

## 안 고른 것

**게임부터 만들고 계약은 나중에 추출** — 흔한 방식이고 대개 맞다. 여기선 아니다.
CLAUDE.md가 게임 타입을 2개로 **영구 고정**했기 때문이다. 계약을 나중에 추출하는 건
"모양을 아직 모른다"는 전제인데, 여기선 이미 안다. 추출을 기다릴 이유가 없다.

**마크다운 명세만 쓰기** — 단계 0이 "코드 거의 안 씀"이니 그럴듯했다. 하지만 마크다운은
컴파일러가 안 읽는다. 타입 드리프트를 못 잡으면 계약을 먼저 정의한 이득이 사라진다.

## 이 결정이 사는 곳

- `packages/protocol/src/*.ts` — 계약 전체. 1,320줄, 로직 0줄
- `packages/protocol/verify.ts` — 계약이 진짜 막는지 확인. `npm run verify`
- `CLAUDE.md` 일정표 단계 0
