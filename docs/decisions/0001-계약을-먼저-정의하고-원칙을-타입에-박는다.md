# 0001. 계약을 먼저 정의하고, 원칙을 타입에 박는다

- 날짜: 2026-07-15 (첫 소비자 검증 반영: 07-16)
- 통합: 옛 0001·0002·0003·0009를 이 파일로 합쳤다 (2026-07-16 문서 경량화)
- 개념: [[스키마-기반-타입-추론]], [[타입-vs-런타임-검증]], [[판별-유니온]], [[상태-머신과-전이표]], [[제네릭과-변성]]

## 무슨 결정인가

세 겹이 한 몸이다.

1. **계약 먼저.** 단계 0은 코드 대신 `packages/protocol/`에 메시지의 모양만 못 박는다. 로직 0줄.
2. **Zod 단일 정의.** 와이어를 건너는 모든 데이터는 Zod 스키마 한 벌로 정의하고 타입은
   `z.infer`로 뽑는다. 예외: `game.ts`는 순수 TS — 와이어를 안 건너는 서버 내부 인터페이스라
   런타임 검증이 낭비다. 검증이 사는 경계는 `parseAnswer(raw: unknown)` 한 지점뿐이다.
3. **원칙을 타입으로 봉인.** CLAUDE.md "변경 금지" 원칙을 주석이 아니라 타입에 건다:
   - `AutoTransition` 2멤버 enum — 앱이 스스로 넘어가는 전이는 딱 둘 (원칙 3)
   - `DisplayRoundView`에 정답 필드가 없다 — 빔 정답 유출이 컴파일 에러 (3-뷰 분리)
   - `DisplayState`에서 BLACK 뺀 전 모드가 `scoreboard` 필수 — "점수판 상시 노출"이 타입

## 왜

- 메시지가 서버·빔·폰·콘솔 **4주체가 만나는 유일한 지점**이다. 계약이 없으면 불일치가
  합치는 날 터진다. 1인 개발이라 더하다 — 리뷰어가 없어서 어긋난 걸 늦게 안다.
- 치팅이 위험목록 1위다("반드시 나온다"). TS 타입은 컴파일 후 사라지므로 런타임 검증은
  선택이 아니고, 두 벌 정의(타입 + 검증 함수)는 반드시 어긋난다. Zod 한 벌이 둘 다 준다.
- **조용히 망하는 것들은 타입으로 막아야 한다. 시끄럽게 망하는 건 어차피 테스트가 잡는다.**
  원칙 위반(자동 진행, 정답 유출)은 조용히 망하는 종류다 — 8시 40분에 주석은 안 읽혀도
  컴파일 에러는 안 읽을 수가 없다.

## 안 고른 것

- **게임부터 만들고 계약은 나중에 추출** — 게임 타입이 2개로 영구 고정이라 모양을 이미 안다.
- **Nest class-validator DTO** — 데코레이터 기반이라 브라우저(빔/폰) 재사용이 안 되고 계약이 두 벌 난다.
- **주석+리뷰 / 런타임 assert / 커스텀 린트** — 리뷰어가 없다(1인 개발). 그리고
  8시 35분에 assert가 터지면 그건 방어가 아니라 사고다. 컴파일 타임에 막아야 무대에 안 오른다.

## 검증 — 첫 소비자가 계약 버그 2개를 찾았다 (옛 0009)

단계 1의 30줄짜리 더미(`dummy.game.ts`)가 `LockRevealGame`을 실제로 구현하자 드러났다:

| 전 | 후 | 왜 |
|---|---|---|
| `reveal: TReveal` (제네릭) | `reveal: ContentChunks` | 코어가 제네릭 리빌을 빔에 보낼 경로가 없었다. 문제도 리빌도 같은 어휘로 그린다 |
| `Game = LockRevealGame<never,…>` | `<unknown, unknown>` | `never`는 바닥이라 아무 게임도 대입 불가 (반환은 공변 위치). 지우개는 천장(`unknown`) |

둘 다 `verify` 전부 통과 중이었다 — 검증은 와이어(Zod)를 보는데 이건 서버 내부(순수 TS)라서.
**계약 먼저가 산 것은 "계약이 완벽하다"가 아니라 버그를 만나는 자리다** — 단계 1 더미에서 20분에 끝났다.
나중에 추출했으면 단계 2에서 게임 코드를 의심하며 헤맸다. 더미는 계약을 진짜로 소비해야 값을 낸다.

단계 2의 첫 소비자(배신 라운드 = `TEAM` scope 첫 사용, 음수 델타 첫 발생)도 같은 종류를 더 찾았다:

| 전 | 후 | 왜 |
|---|---|---|
| `playPrompt: TPlayPrompt` (제네릭) | `playPrompt: PlayPrompt` (Zod 어휘, `play.ts`) | reveal과 판박이 — 와이어를 건너는데 유니온이 제네릭을 지워 폰이 그릴 수 없었다. 덤: "문제 본문은 폰에 안 간다"가 주석에서 어휘로 내려갔다 (본문 실을 필드가 없다) |
| 폰 `mine` = 내 마지막 제출 | TEAM이면 ★우리 조 답★ + 작성자(by) | 안 고치면 배신 라운드에서 폰이 "내 답: 협력"을 띄우는 동안 조 답은 배신 — 적극적으로 틀린 화면 |
| `HEADS_UP`에 scope 없음 | scope 추가 | "내 답"과 "우리 조 답"은 다른 문구다 |
| 델타 UI `+{delta}` 하드코딩 | 부호·색 분기 | 음수가 "+-300"으로 떴다. 음수는 배신 라운드가 처음 만든다 — 첫 소비자 전까지 아무도 모른다 |

단계 3의 첫 소비자(Live 엔진 = 20Hz 우회 채널·별도 상태머신의 첫 사용)는 단계 0에 미리
깔아둔 Live 계약에서 구멍 6개를 찾았다. 미리 깐 계약은 소비 전까지 전부 가설이다:

| 전 | 후 | 왜 |
|---|---|---|
| `DisplayState`에 LIVE 모드 없음 (매치는 `live:armed` 이벤트로만) | `{mode:'LIVE', card, phase, pos, …}` 멤버 | ACTIVE 중 빔이 새로고침되면 복구 불가 — 이벤트는 놓치면 끝이지만 스냅샷은 1발이다. 밤의 피크에 빔이 장님이 되는 구멍 |
| S2C에 `live:armed`·`live:phase`·`live:result` | 삭제 — 우회 채널은 frame(20Hz)·hostTick(2Hz) 2개뿐 | 전이 빈도 정보가 이벤트로 가면 "빔에 뭐가 떠 있나"의 진실이 두 줄기 (0002와 같은 병). 구현 전에 지워서 공짜였다 |
| `TapTugPayload.remainMs` | 삭제 — `{pos}`뿐 | ids.ts "남은 시간은 절대 보내지 않는다" 위반. volatile 프레임이 끊기면 시계가 얼어붙는다 — 타이머는 스냅샷 `phaseEndsAt`에서 파생 |
| `LiveArmSpec.matchId·roundId` (콘솔이 채번) | 서버 채번, 스펙에서 제거 | 멱등 커밋의 키를 콘솔이 발명하면 콘솔 새로고침 후 재사용 → 이미 커밋된 매치와 충돌 → 새 매치 점수가 조용히 증발 |
| `LiveSide.label·tone` | 삭제 — teamId로 점수판에서 파생 | 조 이름·색은 점수판이 이미 전 클라에 준다. 콘솔이 라벨을 지어내는 두 번째 진실 |
| Live 자동 전이가 암묵적 | `LiveAutoTransition` 2멤버 enum + verify §8 | "앱이 스스로 하는 전이" 전수 목록이 라운드 머신만 덮고 있었다 — 매치 머신에도 정확히 2개(GO 착지·매치 종료)를 봉인 |

## 이 결정이 사는 곳

- `packages/protocol/src/` — 계약 전체. `game.ts` 상단 주석이 Zod 경계의 전문
- `packages/protocol/verify.ts` — 원칙이 진짜 박혔는지 실행으로 확인 (`npm run verify`)
- 역대 첫 소비자: 더미(단계 1, 지금은 삭제) → 배신 라운드(단계 2) → Live 엔진(단계 3)
