# 0004. 기록 시스템 규칙 — 아홉 번 결정한 것의 최종형

- 날짜: 2026-07-16
- 통합: 옛 0006·0007·0008·0010·0011·0012·0013·0014·0016. 문서 시스템에 대한 ADR이 코드에
  대한 ADR보다 많아진 것 자체가 병이었다 — 결론만 남기고 아홉을 지웠다. 과정은 git 이력에 있다.

## 규칙

1. **학습 세션은 읽기 전용.** 발견은 `docs/findings.md`에 append하고 개발 세션이 처리한다.
   학습 세션의 컨텍스트엔 설계 의도가 없어 판단 근거가 얇다 — 실제로 `DISPLAY_MODE` 버그를
   정확히 찾고도 틀린 방향으로 고칠 뻔했다. **발견은 싸고 결정은 비싸다.** (옛 0006)

2. **findings는 큐다 — 지시가 아니라 주입.** `CLAUDE.md`가 `@docs/findings.md`로 임포트해서
   매 세션 시작에 내용이 통째로 들어온다. 처리한 항목은 **지운다**(체크가 아니라 삭제 —
   큐지 로그가 아니다). "세션 끝에 본다" 같은 규칙은 넣지 마라 — 끝은 이벤트가 아니다.
   노트북은 그냥 덮힌다. 도중에 쌓인 건 다음 세션 시작이 잡는다. (옛 0013·0014)

3. **개념 노트 태그는 두 축.** `#스텁`(본문 안내줄, 채우면 줄째 삭제) + `영역/…`(frontmatter,
   영구 분류·5개 고정). 티어·하위태그 금지 — `#스텁/당일`은 채우면 목록이 비어서 정작 그 밤에
   쓸모없었다. 그 밤의 지식은 `program-ops.md`가 갖는다. 순서·게이트도 금지 — 학습이 숙제가
   되면 죽는다. (옛 0007·0012)

4. **문서는 재고를 세지 않는다.** "지금 N개" 같은 현재형 개수는 시스템이 잘 돌수록 빨리 썩는다
   (README의 "유일하게 채워진 노트"가 어느새 6개였다). 찾는 법(태그 창, grep 명령)만 적는다.
   상한·예산("영역 5개 고정")은 써도 된다 — 세상을 서술하지 않고 제약하니까. (옛 0008)

5. **개념 노트는 가변, 스텁의 출구는 둘.** 틀린 설명은 고치고, 잘못 꽂힌 스텁은 기존 노트에
   흡수하고 지운다(그 용어를 흡수한 노트 본문에 남길 것 — 안 그러면 다음 세션이 도로 꽂는다).
   사용자가 붙인 태그·메모는 전부 보존한다 — 보관함은 사용자 것이다. (옛 0010·0011)

6. **ADR 불변 규칙은 뺐다 — 이 문서 자체가 그 증거다.** 옛 규칙("본문 불변, 뒤집으면 새 ADR +
   상태 변경")의 결과가 서로를 참조하는 메타 ADR 9개였다. append-only 이력은 git이 이미
   공짜로 준다 — 문서 폴더가 그걸 재구현하면 이력이 두 벌이 된다. 지금 규칙:
   **`decisions/`는 항상 현재 유효한 판단만 담는다.** 뒤집히면 본문을 고치거나 지운다.
   `대상/제품 | 작업방식` frontmatter 축은 유지한다. (옛 0016)

## ADR을 쓰는 기준 (변경 없음)

둘 다 통과해야 쓴다: ① 답이 파일 2개 이상에 걸친다 ② 누가 다르게 했을 법하다.
해당 없으면 코드 주석. 억지로 짜내면 아무도 안 읽는 폴더가 된다 — 실제로 그렇게 됐었다.

## 이 결정이 사는 곳

- `CLAUDE.md` "세션 기록" + `@docs/findings.md` 임포트
- `.claude/commands/learn.md` · `logbook.md` — 각 세션의 실행 규칙 (전문은 저기, 여기는 왜)
- `docs/README.md` — 사람용 사용법
