# 0009. 계약의 제네릭 두 개를 첫 소비자가 무너뜨렸다

- 날짜: 2026-07-16
- 상태: 확정
- 개념: [[제네릭과-변성]], [[스키마-기반-타입-추론]]

## 무슨 결정인가

단계 1에서 계약(`packages/protocol/src/game.ts`)을 두 군데 고쳤다. 둘 다 제네릭이고,
둘 다 **소비자가 0개일 때만 성립하던 타입**이었다.

| 전 | 후 | 왜 |
|---|---|---|
| `ScoreResult<TReveal = unknown>` · `reveal: TReveal` | `reveal: ContentChunks` | 코어가 이 값을 빔에 보낼 경로가 없었다 |
| `Game = LockRevealGame<never, never, never>` | `LockRevealGame<unknown, unknown>` | ★어떤 게임도 대입할 수 없는 타입이었다★ |

그리고 계약에 **빌드가 생겼다**(`tsconfig.build.json`, `exports → dist`).

## 왜 — 1. reveal은 제네릭일 수 없다

`reveal`은 게임이 만들고 빔이 그린다. 그런데 셋이 동시에 참이었다:

- 빔이 받는 `DisplayRoundView.content`는 `ContentChunks`다. 빔은 `DisplayChunk` 말고
  그릴 줄 아는 게 없고, **그게 청크 예산의 존재 이유다**(뒷줄 가독성을 게임마다 다시
  판단하지 않겠다는 것)
- `Game` 유니온이 제네릭을 지운다. 코어가 보는 `reveal`은 `never`다
- `reveal`은 와이어에 자기 표현이 없다 — `events.ts` 어디에도 안 나온다.
  `content`를 통하지 않으면 빔에 도달할 경로 자체가 없다

즉 제네릭인 한 게임이 낸 리빌을 띄울 방법이 없었다. `displayPrompt.content`가 이미
`ContentChunks`였으니 대칭이 맞는다 — **문제도 리빌도 같은 어휘로 그린다.**

## 왜 — 2. 지우개는 바닥이 아니라 천장이어야 한다

`Game = LockRevealGame<never, never, never>`. 레지스트리가 게임을 담으려고 제네릭을
`never`로 지운 것인데, 방향이 반대였다:

```ts
declare const quiz: LockRevealGame<P, string>;
const a: LockRevealGame<unknown, unknown> = quiz;  // 통과
const b: LockRevealGame<never, never> = quiz;      // TS2322: 'P' is not assignable to 'never'
```

`loadRounds`가 `Promise<readonly RoundSpec<TPlayPrompt>[]>`를 반환한다. 반환은 공변
위치라 `RoundSpec<P>`를 `RoundSpec<never>`에 넣으려면 `P`가 `never`에 대입돼야 하는데
그런 타입은 없다. **`never`는 모든 것의 부분타입이지 모든 것의 상위타입이 아니다.**

`unknown`은 반대라 전부 들어간다. 지울 땐 바닥이 아니라 천장으로 지운다.

## 안 고른 것

**빔이 TReveal을 해석하게 한다** — 빔에 게임별 컴포넌트를 두는 안. 이러면
`DisplayChunk` 유니온의 존재 이유가 사라지고("게임 모듈이 자유롭게 HTML을 그리는 게
아니라 이 중에서 고른다"), 게임 추가가 빔 코드 수정이 된다. 계약이 막으려던 바로 그것이다.

**코어에 게임별 렌더러를 둔다** — 코어가 게임을 알게 된다. "레이어로 짠다" 위반.

**`Game` 유니온을 안 쓰고 게임마다 구체 타입으로 분기** — 단계 2에서 게임이 2개가 되는
순간 코어에 `if (gameId === 'quiz')`가 생긴다. 원칙 2가 "코어가 kind로 분기하는 곳은
정확히 한 군데"라고 못 박은 것과 충돌한다.

## 계약에 빌드가 생긴 것

단계 0까진 소비자가 없어서 생 TS를 export해도 됐다(`main: ./src/index.ts`).
서버가 들어오면서 안 된다 — Nest DI는 `emitDecoratorMetadata`를 요구하는데 esbuild/tsx가
그걸 안 하고, tsc로 컴파일하면 런타임에 `node_modules`의 `.ts`를 못 읽는다.
소비자가 둘(Node/브라우저)이라 어느 한쪽 사정에 맞추면 다른 쪽이 깨진다 —
`tsconfig.json`의 `types: []` 주석이 이미 말하던 것과 같은 문제다.

## 이게 0001을 반박하는가 — 아니다, 입증한다

셋 다 "계약을 코드보다 먼저 정의한다"의 실패처럼 보인다. 실제론 반대다.

**셋 다 첫 소비자가 도착한 순간에만 드러나는 종류다.** 타입을 아무리 노려봐도 안 보이고,
`npm run verify` 23개도 전부 통과했다 — 검증이 **와이어 스키마**(Zod)를 보는데 이건
**서버 내부 인터페이스**(순수 TS)라서다. `game.ts`가 Zod가 아닌 이유가 그 자리에 적혀 있다.

0001이 산 것은 "계약이 완벽하다"가 아니라 **버그를 만나는 자리**다.
단계 1에서 30줄짜리 더미를 쓰다 만났고 고치는 데 20분 걸렸다. 계약을 나중에 추출했다면
단계 2에서 진짜 퀴즈를 짜다 만났을 것이고, 그땐 **게임 코드를 의심했을 것이다** —
계약이 원인인 줄 모른 채. 그게 0001이 말한 "합치는 날 터진다"의 정확한 모양이다.

더미가 `LockRevealGame`을 구현한 게 이걸 앞당겼다. 인라인으로 채점했으면 셋 다 단계 2로
넘어갔다. **"게임 0개"는 게임을 안 만든다는 뜻이지 계약을 안 쓴다는 뜻이 아니다.**

## 이 결정이 사는 곳

- `packages/protocol/src/game.ts` — `ScoreResult.reveal`, `Game` 유니온. 둘 다 주석에 이유가 있다
- `packages/protocol/tsconfig.build.json` · `package.json` — dist 빌드
- `apps/server/src/games/dummy.game.ts` — 계약의 첫 소비자. 이 파일이 위를 전부 찾아냈다
- `apps/server/verify-loop.mjs` — 리빌 루프가 실제로 도는지. `npm run verify -w @mt/server`
