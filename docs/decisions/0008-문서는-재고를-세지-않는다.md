# 0008. 문서는 재고를 세지 않는다

- 날짜: 2026-07-16
- 상태: 확정
- 개념: 없음 — 문서 작성 규칙이다

## 무슨 결정인가

안내문에 **"지금 몇 개"를 쓰지 않는다.** 목록과 개수는 태그가 답하고, 문서는 **찾는 법**만 쓴다.

**단, 상한은 쓴다.** 둘의 차이가 이 결정의 전부다:

| | 예 | 왜 |
|---|---|---|
| **재고** — 안 쓴다 | "유일하게 채워진 노트", "현재 당일은 2개다(A, B)" | 시스템이 정상 동작하면 틀려진다 |
| **예산** — 쓴다 | "당일이 5개, 10개가 되면 안 된다", "영역 5개는 고정" | 숫자가 요점이다. 안 쓰면 규범이 사라진다 |

재고는 **세상을 서술**하려다 실패한다. 예산은 **세상을 제약**하므로 실패할 수가 없다.

## 왜

`docs/README.md`가 이렇게 말하고 있었다:

> 1. `concepts/타입-vs-런타임-검증.md` — **유일하게 채워진 노트.** 나머지가 어떤 모양이 될지 보여준다

쓸 때는 맞았다. 발견 시점엔 채워진 게 여섯 개였다. 그리고 이건 **새 사람이 맨 처음 읽는
안내문**이라, 틀리면 그 자리에서 신뢰가 깎인다. 게다가 여섯 중 하나만 가리켜서 나머지
다섯이 안 읽혔다.

**★핵심은 이게 사고가 아니라 정상 동작의 결과라는 것이다.★**

`/learn`이 노트를 채우면 `#스텁` 태그를 뗀다. 그게 이 트랙이 하라고 만든 일이다.
그때마다 "채워진 건 N개"는 조용히 틀려진다. **큐를 비우는 게 문서를 깨뜨린다.**
잘 굴러갈수록 빨리 썩는 문장이라, 아무도 안 고칠 때가 아니라 **잘 쓸 때** 무너진다.

같은 문장이 세 곳에 더 있었다. `README.md`의 "개념 3개", `learn.md`와 `logbook.md`의
"현재 당일은 2개다(원장/fold, websocket/room)". 마지막 둘은 특히 나쁘다 —
`/learn`으로 당일 스텁을 하나 채우면 태그가 떨어져서 당일이 1개가 되는데,
**그 경로엔 문서를 고치라고 알려줄 게이트가 없다.** 당일을 *늘릴* 땐 근거를 대라는
규칙이 있지만, *채우는* 건 그냥 잘한 일이라 아무도 안 멈춘다.

## 안 고른 것

**숫자를 최신으로 고친다 ("유일하게" → "여섯 개 중 하나")** — 자연스러운 수정이고,
아마 대부분 이걸 한다. 그리고 **다음 `/learn` 한 번에 다시 틀린다.** 고치는 게 아니라
만료일을 미루는 것이다. 발견한 학습 세션이 이걸 먼저 짚었다.

**"6개 (2026-07-16 기준)"처럼 날짜를 박는다** — 정직해지지만 안내문이 각주가 된다.
그리고 읽는 사람이 "지금은 몇 개지?"를 스스로 확인해야 하는데, 확인하는 법은 안 알려준다.
그럴 거면 **확인하는 법만 쓰는 게 짧고 항상 맞다.**

**스크립트로 개수를 자동 갱신** — 옵시디언 보관함은 파일이 전부라는 게 이 시스템의 전제고
(`README.md`: "폴더를 읽는 마크다운 뷰어일 뿐이다"), 문서를 빌드 산출물로 만들면 그게 깨진다.
그리고 태그 창이 이미 실시간으로 하는 일이다.

**배너 위치로 센다** — 발견한 학습 세션의 제안이었다. 코드블록 인용에 안 속게, 태그가
아니라 **자리**를 규약으로 삼는다:

```bash
awk 'FNR==3 && /#스텁/ {print FILENAME}' docs/concepts/*.md   # 3번 줄만 본다
```

영리하고, **하루도 못 갔다.** [[0007-스텁-분류는-영역-축을-따로-두고-티어는-2개로-유지한다]]가
영역 태그를 frontmatter로 보내면서 배너가 3번 줄에서 8번 줄로 밀렸다. 지금 저 명령은
스텁 0개 / 채워진 것 19개라고 답한다. **아무도 한 번 써보기 전에 썩었다.**

그리고 이게 이 ADR의 요점을 정확히 반복한다 — 위치를 박는 건 개수를 박는 것과 **같은 종류의
실수**다. 둘 다 "지금 세상이 이렇게 생겼다"를 문서에 복사하는 것이고, 세상은 움직인다.
**마크다운을 제대로 파싱하는 건 옵시디언의 메타데이터 캐시뿐이다.** 텍스트 매처(`grep`,
`awk`, 전문 검색)는 전부 코드블록·frontmatter·줄 위치 중 하나에 걸린다.
세는 건 인덱스한테 시킨다.

## 세는 법 — 텍스트 매칭이 아니라 태그 창

**채워진 노트: `Ctrl+Shift+F` → `path:concepts -tag:#스텁`. 스텁: 태그 창 `스텁`.**

`grep "#스텁"`은 틀린다. `concepts/스텁.md`는 채워진 노트인데 스텁 템플릿을 인용하느라
본문 코드블록에 `#스텁/성장`이 들어 있다. 옵시디언은 코드블록 안의 태그를 안 센다(맞다).
grep은 센다(틀린다). **태그를 다루는 노트는 태그를 인용하게 되어 있으니 이건 계속 생긴다.**

이 함정은 실전이다. 이 규칙을 쓰고 나서 돌린 검증 스크립트가 같은 grep을 써서
`스텁.md`를 스텁으로 잘못 셌다. **규칙을 문서에 적은 직후에 그 규칙을 어긴 것이다.**
사람이 조심해서 피할 수 있는 종류가 아니라는 뜻이다.
