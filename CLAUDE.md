# IGRUS MT 조별 대결 플랫폼

동아리 MT 저녁 프로그램용 실시간 조별 점수 대결 웹앱.
당일 1회 운영이지만 이후 행사 재사용을 전제로 설계한다.

## 스택

- 서버: Nest.js + Socket.IO / 3뷰: React + Vite
- 계약: `packages/protocol` — Zod 단일 정의. 서버와 3뷰가 같은 걸 import한다
- DB: PostgreSQL (문제 은행 + 결과 아카이브 전용). 로컬은 Docker
- 배포: 없음. 사회자 노트북 로컬 실행 + 공유기 LAN

React인 이유: 계약이 역할별 스냅샷이라 3뷰는 **"스냅샷 받아서 그리기"**만 한다. 클라 상태가
거의 없는 게 React의 행복경로다. 바닐라로 하면 그 갱신(diff)을 손으로 재구현하게 된다.

## 성공 기준

**오디오가 비지 않는 것.** 40명이 조용히 폰만 보고 있으면 실패다.
기능이 적어도 소리가 나면 성공이다. 모든 설계 판단은 이 기준으로 한다.

## 핵심 원칙 (변경 금지)

1. **리빌 연출이 본체, 게임은 부품.** 소리는 맞히는 순간이 아니라 **공개되는 순간**에 난다.
   잠금 → 카운트다운 → 동시 공개 → 점수 애니메이션. 즉시 채점하고 넘어가면 프로그램이 죽는다.
2. **게임 타입은 2개뿐. 3번째를 만들지 마라.** LockReveal(수집→잠금→공개→채점) /
   Live(입력 스트림을 빔에 실시간 반영). 어느 쪽에도 안 맞으면 만들지 않는다.
3. **앱은 사회자를 증폭한다. 대체하면 마이너스.** 현장 오디오의 90%는 사회자가 만든다.
4. **실시간 상태는 메모리, 확정 점수는 영속화.** 라운드 확정 점수는 원장에 쌓고 재시작 시
   복구한다. 프로세스가 죽어 점수가 날아가는 게 실제 시나리오다.

## 3-뷰 (물리적으로 분리)

| 뷰 | 라우트 | 대상 |
|---|---|---|
| 빔 | `/display` | 빔프로젝터 전체화면 |
| 폰 | `/play` | 참가자 |
| 콘솔 | `/host` | 사회자 노트북 |

**빔과 콘솔은 절대 같은 화면에 뜨면 안 된다 — 콘솔에 정답이 있다.** 확장 모드 전제.

- 빔 제약(타협 금지): 배경 `#000` + 형광 텍스트, 회색 금지(밝은 방 + 저가 빔에서 날아간다) /
  정보 덩어리 4개 이상 금지(뒷줄 가독) / 점수판 상시 노출(순위 변동이 보여야 야유가 나온다)
- 콘솔 필수기능(현장은 계획대로 안 간다): 라운드 스킵·되감기 / 점수 수동 보정 / 배점 2배 토글 /
  **낮 야외 PG 점수 수동 입력** / 예비 게임 즉시 투입
- **사운드는 필수, 부가기능 아님.** 오디오가 죽는 구간은 게임 사이가 아니라 **입력 대기 10초**다.
  카운트다운 틱 / 긴장 BGM / 리빌 스팅을 빔이 직접 낸다.

## 당일 운영

저녁 8:00 시작 — **8:49 종료** (직후 술자리). 낮 야외 PG + 물놀이 선행이라 참가자는 방전 +
목 상한 상태. 낮 점수를 이어받아 "결승전" 프레임. QR 사전 접속은 밥 먹는 동안.

라인업: ① 낮 점수 공개 3분 ② 내부 퀴즈(LockReveal) 16분 ③ 탭 줄다리기(Live) 12분
④ 배신 라운드(LockReveal) 13분 ⑤ 시상+단체사진 5분.
**예비(정원 1): 번역 릴레이** — 개발 비용 ~0. 사회자가 입으로 채점하고 ADJUST로 점수를 준다.

세부 타임라인·진행 판단·현장 체크리스트는 `docs/program-ops.md`.

## 하지 말 것

- **게임 추가 제안 금지.** 4개 개발 / 당일 3개 확정. 여유가 생기면 게임이 아니라
  **리허설 횟수**에 쓴다 — 리허설 1회가 게임 1개보다 효과가 크다.
- **검토하고 뺀 것들 (재제안 금지):** 데시벨 대결(목 상함 + 8시 민원) / 마피아 자동
  사회자(원칙 3 위반) / 슬라이더·두진실(룰 설명 2분 = 방전 상태에 사망) / AI 실시간
  판정(지연 + 논란 + 오프라인 LAN 충돌) / 무음 게임 일체(성공 기준 위반) /
  "5초 더" 연장 버튼(사망 구간을 늘린다 — `docs/decisions/0002`)
- **클라우드 배포 금지.** 숙소 와이파이는 복불복. QR은 로컬 IP를 담는다. 인터넷 의존 기능 금지.

## 알려진 위험

- **치팅:** IT동아리다. 콘솔로 이벤트 뿌리는 시도가 반드시 나온다. 서버 탭 레이트
  상한(인간 한계 ~15tap/s) + 이상치 감지 필수.
- **조원 수 불균형:** 탭 줄다리기는 대표 3명 제한으로 흡수 (37명이 응원 = 오디오).
- **폰 배터리:** 물놀이 후 8시면 20% 이하 속출. 이탈/재접속(resumeToken) 처리가 실제로 필요하다.
- **배점 밸런스:** 배신 라운드는 2~3위가 1위를 뒤집되 5~6위는 못 뒤집는 수준.
  낮 점수 실측을 보고 SET_POINTS로 맞춘다.

## 일정 (단계별 — 날짜를 박으면 늦어질 때 잘리는 게 리허설이 된다)

| 단계 | 내용 | 완료 조건 |
|---|---|---|
| 0 | 계약 정의 (`packages/protocol`) | 타입체크 + 계약 검증 통과 ✅ |
| 1 | 리빌 루프 + 사운드 + 3뷰 스켈레톤 | 더미 1문제가 소리까지 나면서 완주 ✅ |
| 2 | LockReveal 2개 — ★배신 라운드부터★ | 실제 문제로 퀴즈 8문제 + 배신 1라운드 완주 |
| 3 | Live 엔진 + 탭 줄다리기 | 폰 6대 매치 완주, 레이트 상한 동작 확인 |
| 4 | 콘솔 나머지 (되감기·보정·낮PG·사운드보드·예비 투입) | 콘솔만으로 프로그램 전체 완주 |
| 5 | ★리허설 (동아리원 10명 실제 소집)★ | 타임라인 실측 완료, 자를 것 결정 |
| 6 | 콘텐츠 확정 + 버그 | |
| 7 | 여유가 남으면 예비 1개 (번역 릴레이) | 리허설에 태울 수 있을 때만 |

- **단계 2가 배신 라운드부터인 이유:** 내부 퀴즈는 설문 답장 대기 중(`docs/survey.md`)이고,
  **배신 라운드가 `TEAM` scope의 첫 소비자다** — 미터 분모 분기·TEAM 채점·조 단위 번복이
  아직 한 번도 안 돌았다. 첫 소비자가 계약을 무너뜨린다는 걸 단계 1에서 배웠다(`decisions/0001`).
- **단계 5 도달이 목표다.** 단계 2~4에서 늦어지면 게임을 자르고 리허설로 간다. 반대로 하지 않는다.
- **레이어로 짠다, 게임으로 짜지 않는다.** 게임 하나로 서버·빔·폰을 세로 관통하면 다음 게임에서
  같은 걸 또 짜서 코드가 두 벌 난다. 게임 하나가 30~50줄이면 계약이 제자리, 200줄이면 잘못된 것.

## 개발 외 크리티컬 패스

**회원 설문(내부 퀴즈 문항)을 즉시 돌린다.** 개발 끝내고 콘텐츠를 모으면 답장 기다리다 MT가
온다 — 가장 흔한 사망 원인. 1인 개발에서 유일하게 공짜로 병렬인 작업이다.
문안·필터 기준·목표 응답 수는 `docs/survey.md` — 복붙해서 보내면 된다.
