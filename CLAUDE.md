# IGRUS MT 조별 대결 플랫폼

동아리 MT 저녁 프로그램용 실시간 조별 점수 대결 웹앱.
당일 1회성 운영이 목표지만, 이후 행사 재사용을 전제로 설계한다.

## 스택
- 서버: Nest.js + Socket.IO
- DB: PostgreSQL (문제 은행 + 결과 아카이브 전용)
- 배포: 없음. 사회자 노트북 로컬 실행 + 공유기 LAN

## 성공 기준

**오디오가 비지 않는 것.** 40명이 조용히 폰만 보고 있으면 실패다.
기능이 많아도 소리가 안 나면 실패고, 기능이 적어도 소리가 나면 성공이다.
모든 설계 판단은 이 기준으로 한다.

## 핵심 원칙 (변경 금지)

### 1. 리빌 연출이 제품의 본체다. 게임은 부품이다.
소리는 정답을 맞히는 순간이 아니라 **공개되는 순간**에 난다.
잠금 → 카운트다운 → 동시 공개 → 점수 애니메이션. 이 시퀀스가 코어다.
즉시 채점하고 넘어가면 게임이 아무리 많아도 프로그램이 죽는다.

### 2. 게임 타입은 2개뿐이다. 3번째를 만들지 마라.
- **LockReveal**: 문제 제시 → 입력 수집 → 잠금 → 카운트다운 → 동시 공개 → 채점
- **Live**: 연속 입력 스트림 → 빔에 실시간 반영. 리빌 딜레이 없음.

새 게임은 반드시 둘 중 하나의 인터페이스를 구현한다.
어느 쪽에도 안 맞으면 만들지 않는다.

### 3. 앱이 사회자를 대체하면 안 된다. 증폭해야 한다.
현장 오디오의 90%는 사회자가 만든다. 사회자를 자동화하는 기능은 마이너스다.

### 4. 실시간 상태는 메모리, 확정 점수는 영속화
3시간짜리 세션이다. 실시간 상태를 DB에 넣으면 동기화만 느려진다.
단, **라운드 확정 점수는 매 라운드 스냅샷 저장하고 재시작 시 복구**한다.
로컬 프로세스가 죽어서 점수가 전부 날아가는 게 실제 시나리오다.

## 3-뷰 구조 (물리적으로 분리)

| 뷰 | 라우트 | 대상 |
|---|---|---|
| 빔 뷰 | `/display` | 빔프로젝터 전체화면 |
| 폰 뷰 | `/play` | 참가자 |
| 사회자 콘솔 | `/host` | 사회자 노트북 |

**빔 뷰와 콘솔은 절대 같은 화면에 뜨면 안 된다.** 콘솔에 정답이 있다.
미러링이 아니라 확장 모드 전제로 설계한다.

### 빔 뷰 제약 (타협 금지)
- 배경 `#000` + 형광 텍스트. 회색 계열 UI 금지 — 밝은 방 + 저가 빔에서 전부 날아간다.
- 뒷줄에서 읽혀야 한다. 화면에 정보 덩어리 4개 이상 금지.
- 점수판 상시 노출. 순위 변동이 보여야 야유가 나온다.

### 사회자 콘솔 필수 기능
현장은 100% 계획대로 안 간다. 이게 없으면 프로그램이 죽는다.
- 라운드 스킵 / 되감기
- 점수 수동 보정
- 배점 2배 토글
- **낮 야외 PG 점수 수동 입력** (저녁 프로그램이 이걸 이어받는다)
- 예비 게임 즉시 투입

## 사운드 (필수, 부가기능 아님)
오디오가 죽는 구간은 게임 사이가 아니라 **입력 대기 10초**다.
카운트다운 틱 / 긴장 BGM / 리빌 스팅을 앱이 직접 낸다. 빔 뷰에서 재생.
코드는 몇 줄인데 게임 하나 추가보다 효과가 크다.

## 당일 운영 전제
- 저녁 8시 시작, **8시 49분 종료** (직후 술자리)
- 낮에 야외 PG + 물놀이가 선행. 참가자는 방전 + 목 이미 상한 상태
- 낮 PG 점수를 이어받아 "결승전" 프레임으로 간다
- 참가자는 밥 먹는 동안 QR 사전 접속 (입장 15분 → 5분)

### 당일 라인업 (3개 + 예비)
1. 낮 점수 현황 공개 — 3분
2. 동아리 내부 퀴즈 — LockReveal, 16분
3. 탭 줄다리기 — Live, 12분
4. 배신 라운드(죄수의 딜레마) — LockReveal, 13분
5. 시상 + 단체사진 — 5분

**예비 (여유가 남을 때만. 정원 1개):** 번역 릴레이

번역 릴레이가 예비 1순위인 이유: 개발 비용이 사실상 0이다(채점을 사회자가 입으로 한다 —
`score()`가 델타 0을 반환하고 빔에 조별 답만 띄우면, 사회자가 읽으며 판정하고
점수 수동 보정으로 준다). 룰 설명도 한 문장이고, 설문 데이터도 필요 없다.

## 하지 말 것

### 게임 추가 제안 금지
**4개 개발 / 당일 3개 사용**이 확정이다. 여유가 생기면 게임이 아니라
**리허설 횟수**에 쓴다. 리허설 1회 추가가 게임 1개 추가보다 효과가 크다.

(2인 시절엔 7개 개발이었다. 인원이 반이 되면 게임이 먼저 떨어진다 —
리허설·리빌 루프·사운드·설문은 자를 수 없기 때문에 산술적으로 남는 게 게임뿐이다.)

### 이미 검토하고 뺀 것들 (다시 제안하지 마라)
- **데시벨 대결**: 물놀이로 목이 이미 상함 + 실내 8시 민원
- **마피아 자동 사회자**: 사회자를 없애는 기능. 원칙 3 위반
- **슬라이더 / 두진실**: 8시 방전 상태에 룰 설명 2분은 반응이 죽음.
  2인 시절엔 "예비로만"이었는데, 1인 개발로 예비 정원이 1개가 되면서 완전히 빠졌다.
  룰 설명이 길다고 이미 깎인 둘이 정원 경쟁에서 먼저 떨어진 것이다.
- **AI 실시간 판정 게임** (프롬프트 배틀, 탈옥 방어전 등): API 지연 + 판정 논란 + 오프라인 LAN과 충돌
- **무음 게임** (HEX 색상, 1→50 터치, 커밋 메시지 퀴즈, 코드 결과 맞추기): 성공 기준 위반. 뒤 두 개는 신입생 소외까지

### 클라우드 배포 금지
숙소 와이파이는 복불복이다. 사회자 노트북 로컬 + 공유기 LAN이 기본이고
QR은 로컬 IP를 담는다. 인터넷 의존 기능을 넣지 마라.

## 알려진 위험
- **치팅**: IT동아리다. 탭 줄다리기에서 콘솔로 이벤트 뿌리는 시도가 반드시 나온다.
  서버에서 탭 레이트 상한(인간 한계 ~15tap/s) + 이상치 감지 필수.
- **조원 수 불균형**: 탭 줄다리기는 인원수에 정비례. 대표 3명 제한 또는 인원수 정규화.
- **폰 배터리**: 물놀이 후 8시면 20% 이하 속출. 이탈/재접속 처리가 실제로 필요하다.
- **배점 밸런스**: 배신 라운드는 2~3위가 1위를 뒤집을 수 있되 5~6위는 못 뒤집는 수준.
  크면 "앞에 왜 했냐", 작으면 피날레가 무의미.

## 일정 (단계별, 1인 개발)

주차로 안 쓴다. 혼자면 어디서 막힐지 예측이 안 되고, 날짜를 박아두면 늦어질 때
잘리는 게 리허설이 된다. 각 단계는 **완료 조건**으로 판단한다.

| 단계 | 내용 | 완료 조건 |
|---|---|---|
| 0 | WS 이벤트 스키마 + 게임 인터페이스 2종 계약 정의 | `packages/protocol` 타입체크 + 계약 검증 통과 ✅ |
| 1 | **리빌 루프 + 사운드.** 코어 + 3뷰 스켈레톤 + 콘솔 진행 명령 | 더미 문제 1개로 잠금→카운트다운→동시공개→점수 애니메이션이 **소리까지 나면서** 돈다. 게임 0개 |
| 2 | LockReveal 게임 2개 (내부 퀴즈, 배신 라운드) | 실제 문제로 퀴즈 8문제 + 배신 1라운드 완주 |
| 3 | Live 엔진 + 탭 줄다리기 | 폰 6대로 매치 완주. 탭 레이트 상한 동작 확인 |
| 4 | 콘솔 나머지 (되감기, 점수 보정, 낮 PG 입력, 사운드보드, 예비 투입) | 콘솔만으로 프로그램 전체 완주 가능 |
| 5 | **리허설 (동아리원 10명 실제 소집)** | 타임라인 실측 완료. 뭘 자를지 결정됨 |
| 6 | 콘텐츠 확정 + 버그 | |
| 7 | 여유가 남으면 예비 1개 (번역 릴레이) | 리허설에 태울 수 있을 때만 |

**단계 1에 게임이 0개인 건 의도된 것이다.** 더미 문제로 리빌 연출부터 완성한다.

**사운드가 단계 1에 붙어 있는 것도 의도다.** 소리 없는 리빌 루프는 리빌 루프가 아니다.
뒤로 미루면 늦어질 때 제일 먼저 잘리는데, 그건 "필수, 부가기능 아님"이라고 쓴 것과 모순이다.
잘릴 수 없는 것과 같은 단계에 묶어둔다.

**단계 5(리허설)에 도달하는 게 목표다.** 단계 5를 못 하고 당일을 맞는 게 최악의 시나리오다.
단계 2~4에서 늦어지면 **게임을 자르고 리허설로 간다.** 반대로 하지 않는다.

### 레이어로 짠다, 게임으로 짜지 않는다

2명일 땐 이게 분담 규칙이었다(A=서버·콘솔 / B=빔·폰·연출·사운드). 1인이 되면서 분담은
없어졌지만 **순서 규칙으로 남는다.**

게임 하나를 골라서 서버·빔·폰을 세로로 관통하면, 다음 게임에서 같은 걸 또 짜게 되고
코드가 두 벌 나온다. 혼자여도 똑같다 — 오히려 리뷰해줄 사람이 없어서 두 벌이 됐다는 걸
늦게 안다. 단계 1에서 리빌 루프를 레이어로 완성하고 그 위에 게임을 얹는다.

게임 하나가 30~50줄이면 계약이 제자리에 있는 것이고, 200줄이면 뭔가 잘못된 것이다.

### 개발 외 크리티컬 패스 — ★단계 0과 동시에 시작★

**회원 설문을 지금 돌려야 한다.** 내부 퀴즈는 사전 데이터가 필요한데 사람들이 답을 안 보낸다.
개발 끝내고 콘텐츠 모으면 답장 기다리다 MT 온다. 이게 가장 흔한 사망 원인이다.

**1인 개발이면 이게 더 중요해진다.** 혼자면 개발에 병렬이 없다. 설문은 유일하게
공짜로 병렬로 도는 작업이다 — 보내놓고 코드 짜는 동안 답장이 쌓인다.
단계 순서와 무관하게 **오늘 보낸다.**

## 세션 기록 — 시키지 않아도 한다

개발 세션에서 의미 있는 판단을 내렸으면 **사용자가 명령을 치지 않아도** 남긴다.
규칙 전문은 `.claude/commands/logbook.md`, 시스템 설명은 `docs/README.md`.

- `docs/decisions/NNNN-<slug>.md` — 왜 이렇게 짰나
- `docs/concepts/<개념>.md` — 새로 등장한 개념. **스텁만.** 설명은 사용자가 `/learn`으로 물을 때

**사용자가 기억해서 명령을 쳐야 하면 그 트랙은 죽는다.** 설문이 죽는 것과 같은 이유다.
세션은 깔끔하게 안 끝난다 — 피곤해서 노트북을 덮는다. 그때 재료가 이미 남아 있어야 한다.

**쓰는 기준(둘 다 통과해야 함):** 답이 파일 2개 이상에 걸치는가 + 누가 다르게 했을 법한가.
해당 없으면 **안 쓴다.** 세션마다 하나씩 짜내면 아무도 안 읽는 폴더가 된다.
한 곳에서 끝나는 이유는 코드 주석으로 족하다.
